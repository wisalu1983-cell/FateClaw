# Agent圣杯战争提案-对话提炼

> 用途：沉淀当前对话里已经形成的共识，作为后续继续思考、补规则、拆系统和讨论MVP的底稿。
>
> 说明：本文档基于本轮对话提炼，不等同于最终设定案。

---

## 一、游戏设计创意的核心思路

### 一句话版本

玩家作为 `Master`，拿起现实世界中任意一个手边物品，把它当作“圣遗物”完成召唤。系统会用“大模型 + 规则框架”生成一个独一无二的 `Servant`。这个 `Servant` 会以 agent 的形式，在一座按圣杯战争规则持续运转的小镇里行动。玩家则通过类似 `OpenClaw` 的外部 IM 交互方式，与这个从者沟通、协作、下令，并一起经历侦查、结盟、背叛、战斗与局势演化。

### 更大白话的版本

这款游戏可以先理解成：

玩家和一个会自己想事情、也会自己行动的 `Servant` 结成搭档，一起被丢进一座会自己运转的小镇，去经历一场圣杯战争。

玩家更接近真正的 `Master`：和自己的从者聊天、协商、下令、磨合关系。技能释放层面的直接手操并不是第一层重点。与此同时，小镇里的其他主从也都在自己行动，所以整场战争会自己长出不同局面和故事。

### 核心构成

1. **这是一个 agent 技术驱动的圣杯战争模拟游戏**

   - 重点在于把圣杯战争规则真正做成一个可运行、可推演、可涌现的游戏系统。
2. **召唤的起点来自现实输入**

   - 玩家可以拍摄任意手边物品，把它当作圣遗物。
   - 系统根据输入内容，由规则框架负责硬判定，由大模型负责软生成，得到独一无二的从者结果。
3. **Servant 是可持续行动的 AI agent**

   - 生成结果不仅有职阶、属性、性格、技能方向和立绘，也有行为倾向、决策方式、关系演化和长期行动能力。
   - 从者会思考、判断、协商、试探，也可能误判、抗命或改变关系。
4. **玩家是战争中的 Master**

   - 玩家通过类似 `OpenClaw` 的方式，与从者互动、下令、协商、调查和设局。
   - 玩家和从者的关系本身就是玩法核心，操作单位只是其中一层。
5. **舞台是一座“活着的小镇”**

   - 底座接近“斯坦福小镇 / aivilization”式的持续运行社会模拟。
   - 小镇中的御主、从者、普通人和关键势力都能产生观察、传播、关系变化和策略调整。（备注：我倾向于玩家自身不在世界里。但需要设计在这个设定基础上，玩家可被攻击的机制。后续待定）
6. **`token` 直接设定为魔力（备注：这一条需要再讨论下，哪些行为需要用token，哪些用传统的游戏内数值资源。例如下指令给servant目标让agent自己拆解决策肯定是要用token的，但战斗中开个宝具本质上是处罚一个战斗技能，用传统的魔法值就可以了）**

   - 魔力会直接进入侦查、行动、战斗、令咒式强制控制、恢复和维持现界等核心环节。
   - 玩家每一步都在做战争预算管理。
7. **整体体验重心是“活过一场由 AI 驱动的圣杯战争”**

   - 体验重心落在“你召唤出了谁、如何与其协作、你们如何在一场不断演化的战争中走到最后”。

## 二、目前已讨论的话题与阶段性结论

### 1. 提案定位

- **已讨论内容**
  - 这个创意究竟是什么类型的游戏提案。
  - 玩家、从者、小镇、OpenClaw、agent 各自处在什么位置。
- **阶段性结论**
  - 这是一个“agent 驱动的圣杯战争模拟游戏提案”。
  - 其核心由“召唤生成 + 主从协作 + 小镇演化 + 圣杯战争规则”四层共同构成。

### 2. 对外一句话表达与面向玩家的卖点

- **已讨论内容**
  - 如何向玩家快速说明这是个什么游戏。
  - 面向 TapTap、小红书等场景该强调哪些信息。
- **阶段性结论**
  - 最值得强调的卖点有五个：
    - 任意物品拍照召唤，仪式感强。
    - 召唤出来的是独一无二的 Servant，不依赖固定角色池。（备注：未必完全不依赖，尤其是原型测试时，具体细节待定）
    - Servant 是会行动和博弈的 AI agent。
    - 玩家通过类似 `OpenClaw` 的 IM 式交互，以 Master 身份直接协作和下令。
    - 小镇中的圣杯战争会自然演化，每局故事都不同。

### 3. 召唤系统

- **已讨论内容**
  - 拍照生角色的具体含义。
  - 规则框架与大模型分别负责什么。
- **阶段性结论**
  - 输入源确定为“任意手边物品”，并被视为圣遗物。
  - 规则层负责职阶、属性、阵营、技能方向、性格底盘等硬判定。
  - 大模型负责背景细节、行为风格、说话方式、形象立绘等软生成。
  - 目标是让结果既有神秘感，又能回头讲通，不让玩家觉得是纯随机胡编。（备注：总体上基于fate系列通过圣遗物召唤英灵的设定，还原这一过程的体验）

### 4. 玩家位置与 OpenClaw

- **已讨论内容**
  - 玩家到底像操盘者、旁观者，还是战争中的真实参与者。
  - `OpenClaw` 在玩法里承担什么职责。
- **阶段性结论**
  - 玩家被明确定位为 `Master`，以战争参与者视角行动。
  - 这里提到 `OpenClaw`，主要是为了描述一种交互形态：玩家可以通过外部 IM 式工具与游戏环境中的 agent 互动。
  - 原型测试阶段可以考虑直接部署 `OpenClaw` 来验证这种体验，正式产品未必沿用同一套框架。
  - `OpenClaw` 的价值集中在承接 Master 级行为：
    - 情报搜集
    - 计划制定
    - 资源管理
    - 对从者下达自然语言命令
    - 在关键时刻强制改写局面

### 5. 为什么这个创意容易被误解

- **已讨论内容**
  - 在和朋友解释过程中，这个创意先后被理解成 `moba`、`gal`，说明当前表述方式容易把人带向已有品类印象。
- **阶段性结论**
  - 当描述先从“多组主从对抗、小镇舞台、战前战中策略、资源管理、局势演化”切入时，听者很容易先联想到 `moba`、自走棋或其他对战框架。
  - 当描述后面又转向“和 `Servant` 相处、分支剧情、补魔、关系变化”时，听者又会把它归到 `gal`、视觉小说或关系养成方向。
  - 这说明当前描述里，玩家的真实体验重心没有被放在最前面。
  - 更容易被理解的解释顺序应该是：
    - 先说玩家的身份：玩家是 `Master`，会和一个能自主行动的 `Servant` 搭档。
    - 再说互动方式：玩家通过对话、协商、下令和强制命令影响从者，技能释放的直接手操不放在第一层。
    - 最后再说舞台：这对搭档会被放进一座持续运转的小镇，小镇里的其他主从也都在自己行动。
  - 用于第一次向别人解释时，更稳的说法可以是：
    - 这是一款让玩家和一个会自己思考、自己行动的 `Servant` 搭档，在一座会自己运转的小镇里经历圣杯战争的游戏。

### 6. 引入 agent 后，相对过去 Fate 游戏的关键体验差异

- **已讨论内容**
  - 对方质疑：agent 相对配置更复杂的行为树 NPC，除了“更高级”之外还有什么本质变化。
  - 对话中进一步澄清了 agent 在动态环境、自主决策、持续对话和关系演化上的意义。
- **阶段性结论（备注：归根结底，引入agent后我希望玩家在游戏中能体验到自己是在和一个有独立思考能力、自主行动能力的对象合作，原汁原味地还原FATE正统作品中master和agent之间的关系。）**
  - 这次引入 agent 后，最重要的变化落在主从关系和战争过程都可以持续互动、持续变化，而不再主要依赖预写脚本和固定反馈。
  - 过去 Fate IP 游戏里，玩家面对的对象大多以固定台词、固定分支、固定触发条件为主，很多互动本质上仍然是“和一个机械系统打交道”。
  - 引入 agent 后，`Servant` 可以在动态复杂环境中感知局势、记住过去、做计划、调用能力、反思结果，并据此继续行动。
  - 行为树 AI 可以在预设逻辑里稳定执行动作；agent 的意义在于面对动态环境、开放对话和复杂关系时，依然能持续判断并调整行为。
  - 这会直接改变玩家与从者的相处体验：玩家说的话不再只是触发一个预设回应，而是会进入对方的判断过程。
  - 普通指令的执行结果，会受到从者的性格、目标、当前关系、局势判断等因素影响。
  - 令咒类指令则可以作为高权限、强制执行的特殊机制，和普通指令形成明显区别。
  - 从体验层看，这个提案最关键的新增价值有两点：
    - 过去很难还原的“Master 与 Servant 协作、磨合、博弈”的过程，现在有机会被做成真正的玩法。
    - 圣杯战争的推进过程也能从预定流程，转向一场由多方 agent 持续推动的动态战争。

### 7. 场景与世界骨架

- **已讨论内容**
  - 战争发生在什么样的空间里。
  - 为什么选择“斯坦福小镇 / aivilization”式底座。
- **阶段性结论**
  - 当前舞台方向确定为一座“以圣杯战争规则运转的斯坦福小镇”。
  - 小镇既是日常空间，也是战争容器。
  - 用大白话解释时，可以把“斯坦福小镇”理解成一个把一群 AI 小人放进去后会自己运转的数字空间，接近“模拟人生里的小人活了过来，并慢慢形成一个小社会”。
  - 为了便于理解，也可以暂时把这些 agent 看成“虚拟玩家”，它们会各自判断、行动，并驱动自己的角色在小镇里推进局势。
  - 日常作息、空间移动、社会传播，本身会被转译成圣杯战争玩法。

### 8. 小镇如何承载圣杯战争规则

- **已讨论内容**
  - 如何把圣杯战争规则真正嵌进小镇，避免只停留在背景贴皮层面。
- **阶段性结论**
  - 小镇地点需要被赋予战争功能，例如灵脉点、情报点、伏击点、安全会面点、高曝光风险点等。
  - 昼夜循环要被转成战争节奏：
    - 白天偏调查、伪装、经营关系
    - 黄昏偏试探、跟踪、接头
    - 夜晚偏交战、突袭、结界
    - 战后偏善后、隐藏痕迹、重新分配魔力
  - 普通居民同时承担目击者、舆论传播者和后果放大器的作用。

### 9. 圣杯战争共有规则与 Fate 系列调研（备注：这部分根据\圣杯战争原型设定集 里的资料，再核实一遍）

- **已讨论内容**
  - Fate 各版本中，哪些是圣杯战争最稳定的共通底盘。
  - Master 和 Servant 各自典型会做哪些行动。
- **阶段性结论**
  - 跨版本相对稳定的共通骨架包括：
    - 围绕“愿望机 / 圣杯使用权”的最终目标
    - `Master + Servant` 的契约参战单位
    - 英灵被装入职阶容器现界
    - 令咒作为硬控制与强化机制
    - 围绕情报、真名和身份展开的博弈
    - 持续淘汰直到终局
    - 主从双方都拥有自己的愿望和主观意志
  - Master 的典型行为包括召唤、调查、谈判、结盟、用令咒推进战局、与从者协商计划。
  - Servant 的典型行为包括战斗、使用宝具与技能、独立行动、短暂合作、隐藏或暴露真名相关信息。

### 10. Moon Cell / Fate Extra 的参考价值

- **已讨论内容**
  - Moon Cell 这一代作品的设定、玩法循环，以及它是否适合作为原型。
- **阶段性结论**
  - Moon Cell 的强项在于“准备期 - 情报战 - 决战”的结构化循环，以及主从共同生活、共同作战会议的关系设计。
  - 它更适合作为**玩法节奏参考**，不适合作为“小镇社会模拟”的直接世界模板。
  - 它最值得借用的是：调查与准备优先、情报决定生存、主从关系是持续互动关系。

### 11. 哪个 Fate 版本最适合做“小镇式圣杯战争”原型

- **已讨论内容**
  - 冬木式、Moon Cell、Apocrypha、strange Fake 哪种模式更适合斯坦福小镇。
- **阶段性结论**
  - 单一原型里，**最适合的是冬木式圣杯战争**。
  - 如果允许混搭，当前最优判断是：
    - **世界骨架用冬木式**
    - **玩法节奏吸收 Moon Cell**
    - **后续扩展可参考 strange Fake**
  - `Apocrypha` 更偏阵营大会战，与当前小镇原型的贴合度较低。

### 12. Servant 战斗该如何承载

- **已讨论内容**
  - 斯坦福小镇模式是否适合直接承载从者战斗。
  - 是否需要换一个独立平台来做战斗。
- **阶段性结论**
  - 不需要换成完全独立的平台。
  - 需要的是“小镇社会层”和“战斗解析层”的分层。
  - 小镇层负责让冲突自然发生，战斗层负责把冲突规则化结算，再将结果回写到小镇。

### 13. 最早期原型的战斗表现形式

- **已讨论内容**
  - 早期版本是否要做复杂战斗演出。
- **阶段性结论**
  - 当前已经收敛到一个更轻的原型方向：
    - 战斗先不做高细度演出。
    - 采用“早期 SLG 风格的对战日志 + 结算结果描述”。
  - 玩家在早期原型里主要看到：
    - 为什么打起来
    - 过程中的关键动作日志
    - 伤势、撤退、暴露、地点破坏、目击与舆情等结果
  - 这个方向最适合先验证“agent 驱动的小镇能否自然长出像圣杯战争的局面”。

### 14. MVP 方向

- **已讨论内容**
  - 第一版应该做到什么程度，先验证什么。
- **阶段性结论**
  - MVP 不需要一开始就做完整冬木市、七骑大战或复杂演出。
  - 早先形成的方向是：小地图、小规模参战组数、可运行召唤系统、可解释的规则映射、昼夜循环、侦查 / 协商 / 战斗 / 令咒等核心机制优先。
  - 目前又进一步收敛出一个更轻的落地方向：先把“小镇层 + 战斗日志结算层”跑通。

## 三、目前已经识别出的关键设计空白

以下问题在对话中已经被明确识别，但尚未被定稿，适合作为下一轮重点讨论对象：

1. **召唤映射的可解释性**

   - 如何让玩家觉得“为什么是这个职阶、这个性格、这个能力”是有理可循的。
2. **agent 自主性与游戏可控性的边界**

   - 哪些部分可以自由演化，哪些必须受规则强约束。
3. **小镇地点功能表**

   - 每个地点在圣杯战争里承担什么作用，如何影响侦查、补魔、会面、战斗与暴露。
4. **魔力收支规则**

   - `token = 魔力` 的获得、消耗、透支、恢复和上限逻辑还需要定清楚。
5. **情报暴露规则**

   - 真名、职阶、技能特征、Master 身份分别如何暴露，以及如何被误导或伪装。
6. **冲突升级规则**

   - 从“试探 / 跟踪 / 接触”到“遭遇 / 正式交战 / 宝具释放”的升级条件还需要结构化。
7. **OpenClaw 的玩法边界**

   - 它到底开放到什么程度，哪些能力属于 Master 的核心权能，哪些不应该交给玩家。

## 四、当前可作为后续讨论的总判断

目前已经形成的整体判断如下：

- 这个提案当前最成立的方向，是“由 agent 技术驱动、在活的小镇中自然演化的一场圣杯战争”。
- 其最有辨识度的组合是：
  - **任意物品拍照召唤**
  - **独一无二的 Servant agent**
  - **玩家作为 Master 用 OpenClaw 协作**
  - **token 直接等于魔力**
  - **斯坦福小镇 / aivilization 式社会模拟承载战争外层**
  - **轻量战斗日志系统承载战争结算**
- 现阶段最合理的推进方式，是继续把这个提案从“概念成立”推向“规则表成立”。
