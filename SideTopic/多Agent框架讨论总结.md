# 与游戏玩法解耦的多Agent框架相关思考

## 1. 文档目的

重点讨论两件事：

- 如何定义一个与具体玩法解耦、可通过配置或必要二次开发接入具体游戏玩法的多 agent 框架
- 在 `Client / Agent Framework / Server(Runtime)` 这套结构下，三者的职责边界和主信息流是什么

## 2. 与具体游戏玩法解耦的多Agent框架定义

### 2.1 定义出发点

从 agent 的基本定义出发，agent 是一个能够在环境中感知、思考、决策并行动的运行单元。

在本讨论中，agent 的工作环境是一个特定的游戏环境。因此，这个框架讨论的是让 agent 在具体游戏世界中运行起来的通用机制。

### 2.2 框架定义

多 agent 框架是一个嵌入在游戏 `Runtime` 中的、与具体玩法解耦的角色决策与协作子系统。

更完整地说，它是一个面向特定游戏环境的通用 agent runtime。它为游戏中的各类 agent 提供统一的感知、思考、决策、行动与协作机制，使多个 agent 能在具体游戏世界中持续运行并相互作用。

面向开发者时，可以将其理解为：开发者通过配置或必要的二次开发，把具体游戏的环境与角色信息接入框架；框架负责驱动 agent 持续运行 `Observe -> Think -> Decide -> Act` 循环，并输出行动意图、协作行为与自然语言反馈。

该框架聚焦于角色决策与协作运行，具体玩法规则由游戏侧提供，客户端渲染由 `Client` 承担，最终权威世界状态由 `Runtime` 持有。

### 2.3 接入具体游戏玩法时需要提供什么

接入具体游戏玩法时，需要向框架提供一套“游戏环境适配模型”，用于回答：

- 这个 agent 是谁
- 它处在什么环境里
- 它能看到什么
- 它能做什么
- 它想达成什么
- 它受到什么约束
- 它与谁存在什么关系

进一步归类后，接入内容可分为四类：

- 身份定义：角色设定、能力边界、记忆、人格、权限
- 环境定义：世界状态、时间、地点、资源、事件、可观测信息
- 行为定义：动作接口、动作参数、前置条件、执行反馈
- 决策约束定义：目标体系、关系网络、组织结构、规则限制、胜负条件

### 2.4 与 OpenClaw 的类比

可以将这套框架理解为一种“运行在游戏环境中的 OpenClaw 式 agent 框架”。

- 两者在底层运行逻辑上高度相似，核心都是让 agent 在特定环境中持续执行 `Observe -> Think -> Decide -> Act` 循环
- 两者的核心差异在于运行环境
- `OpenClaw` 面向的是用户电脑办公环境中的任务、软件和信息流
- 本框架面向的是具体游戏环境中的角色、规则、状态和行动空间

因此，更准确的说法是：这套框架的底层运行逻辑与 `OpenClaw` 高度相似，其运行环境是一个特定的游戏环境。

## 3. 工作定位

### 3.1 Agent Framework 的定位

`Agent Framework` 是游戏运行时中的“角色智能层”，其职责是：

- 承载单 agent 的感知、记忆、目标、推理和决策循环
- 承载多 agent 之间的通信、委托、主从协作、联盟协作
- 输出结构化行动意图、角色表达内容和必要的信息请求

其职责边界集中在：

- 角色智能运行
- 多 agent 协作编排
- 结构化行动与表达输出

### 3.2 框架本体包含什么

在将具体游戏玩法的接入配置抽离后，框架本体保留的是一套通用的 agent 运行机制，主要包括：

- 生命周期引擎：驱动 `Observe -> Think -> Decide -> Act` 循环
- 上下文与记忆管理：组织观察结果、历史事件、短期与长期记忆
- 决策编排器：将目标、约束、记忆和当前状态组织成可执行决策
- 多 agent 通信与协作机制：支持消息传递、委托、汇报、主从协作与联盟协作
- 行动提案与结果回接机制：标准化输出行动意图，并接收 `Runtime` 的执行反馈
- 模型与推理适配层：对接具体 `LLM`、推理策略、缓存、重试与成本控制
- 可观测与调试能力：记录日志、决策链路、状态快照与回放信息

### 3.3 与业务解耦的含义

“与具体玩法解耦”在这里指的是：

- 框架通过标准化接口接收具体游戏的玩法规则
- 框架只依赖标准化的观察、动作、约束和反馈接口
- 接入新游戏时，优先通过 `adapter + config` 完成接入

## 4. 三层结构图

```text
┌──────────────────────────── Client ────────────────────────────┐
│ 职责：用户输入输出、UI渲染、地图表现、日志与对话展示             │
└───────────────┬───────────────────────────────┬────────────────┘
                │ 用户输入 / 操作请求            │ 状态更新 / 可见事件 / 文本
                │                                │
                v                                ^
      ┌──────────────────── Server / Runtime ────────────────────┐
      │ 职责：权威状态、时间推进、调度、事件分发、最终结果提交      │
      │                                                           │
      │  ┌───────────────┐        ┌───────────────────────────┐   │
      │  │ 代码规则判定   │        │ LLM裁判判定               │   │
      │  │ 合法性/数值/   │        │ 开放式结果判断             │   │
      │  │ 资源/时间结算  │        │ 态度/许可/说服等语义判定   │   │
      │  └───────┬───────┘        └──────────────┬────────────┘   │
      └──────────┼───────────────────────────────┼────────────────┘
                 │ 观察包 / 状态 / 目标 / 约束    │ 行动意图 / 对话 / 请求
                 v                               ^
        ┌──────────────── Agent Framework ────────────────┐
        │ 职责：角色决策、多agent协作、结构化行动输出       │
        └──────────────────────────────────────────────────┘
```

## 5. 主信息流

标准主链路如下：

1. `Client` 提交玩家输入
2. `Server/Runtime` 基于当前权威状态决定是否调用 `Agent Framework`
3. `Agent Framework` 返回结构化行动意图、对话内容或信息请求
4. `Server/Runtime` 通过代码规则或 `LLM` 裁判完成校验与结算
5. `Server/Runtime` 更新权威状态
6. `Server/Runtime` 将可见结果同步给 `Client`，并将新状态反馈给相关 agent

## 6. 当前结论

- `Client` 负责面向用户的输入输出
- `Agent Framework` 负责角色智能与协作，并向 `Runtime` 输出结构化行动与表达
- `Server/Runtime` 负责权威状态与调度，是系统中枢
- 最终结果可以来自代码规则判定，也可以来自 `LLM` 判定
- 所有结果统一经由 `Runtime` 提交到权威状态

## 附录A：边界说明

### A.1 Agent 的 `Act` 应如何理解

`Act` 更适合被定义为“agent 对外发出的结构化动作”。

常见输出包括：

- 行动意图，如移动、侦察、攻击、协商
- 角色表达内容，如汇报、发言、态度反馈
- 信息请求，如请求额外观察、请求当前可执行动作

### A.2 Agent 输出的主去向

推荐的主链路是：

- `Agent Framework` 主要与 `Runtime` 交互
- `Runtime` 负责提交状态、分发结果和控制信息可见性
- `Client` 负责接收可见结果并完成展示

### A.3 权威结果的来源

“权威”描述的是由谁负责提交状态。当前框架支持两类结果来源：

- 代码规则确定结果，`Runtime` 提交状态
- `LLM` 裁判判定结果，`Runtime` 提交状态

这种设计同时支持强规则游戏和开放式语义判定游戏。
