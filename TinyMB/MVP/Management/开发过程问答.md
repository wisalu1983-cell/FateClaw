## 一、MVP 方法与原型边界

### QA-01 什么是“桩”？

- **问题**：`宏循环（单局/单天）— MVP 中可为桩` 里的“桩”是什么意思。
- **澄清结论**：这里的“桩”指最小占位实现。首轮原型不需要做完整长期循环，只要先证明“世界在演变、玩家介入会留下后果”即可。
- **影响**：宏循环可以先收束为 `7~14 天游戏内时间` 的短周期验证片段，完整城主线、长期政治演化和完整终局后放。

### QA-02 命名 NPC 和非命名角色有什么区别？

- **问题**：命名 NPC 在游戏版本中的定位是什么，和非命名 NPC 有什么本质差异。
- **澄清结论**：命名 NPC 是完整 Agent 实体，具备唯一名字、性格、目标、派系身份和完整 `Observe -> Think -> Decide -> Act` 循环；非命名角色只是环境描述或规则触发器，不跑独立 Agent 循环。
- **影响**：首轮原型中“世界活着”的核心承载者是 `6 名命名 NPC`；背景人群只需文本存在，不进入完整 Agent 成本。

### QA-03 四级自主等级与完整请示节奏是什么意思？为什么首轮不全做？

- **问题**：四级自主等级与完整请示节奏具体指什么，为什么不在首轮原型里全部实现。
- **澄清结论**：完整母稿包含“完全自主 / 事后通知 / 建议后等待确认 / 必须等待玩家拍板”四级；首轮原型只保留最小基线：`默认自主 + 重要决策请示 + 未回复安全回退`。
- **影响**：首轮重点验证“玩家引导感”和“能力差异是否可感知”，不提前把交互层级做得过细，避免信息负载和实现复杂度膨胀。

### QA-04 回标前的 5 篇设计文档是否需要推倒重来？

- **问题**：阶段 0/1 明确后，之前先写出来的 5 篇设计文档是否已经失效，需要从头设计。
- **澄清结论**：不需要全部推倒重来。它们属于阶段 2 的先行产物，问题在于前置信息变了，需要按正式的体验假设和范围文档进行 `MVP-Must / Later` 回标和局部重写。
- **影响**：后续采用“回标 + 收边 + 局部改写”的方式，而不是完全重建，保留已有有效设计资产。

### QA-05 “首轮可验证原型”和“稍完整的 MVP 首版”差别是什么？

- **问题**：首轮可验证原型与稍完整的 MVP 首版在设计目标上差在哪。
- **澄清结论**：首轮可验证原型只负责验证体验假设是否成立，要求最小可玩、最小可观察、最小可比较；稍完整的 MVP 首版才会补更多区域、更多 NPC、更完整政治层和更细交互节奏。
- **影响**：首轮原型锁为 `4 个活跃区域 / 6 名命名 NPC / 轻量外部压力 / 最小请示基线`，后续再向更完整首版扩展。

### QA-06 外部威胁在游戏循环中起什么作用？

- **问题**：轻量外部威胁到底在循环里承担什么功能，为什么需要它。
- **澄清结论**：它的作用不是直接提供完整战争系统，而是为四区行动持续注入新的情报源、任务源和短期站队压力，避免世界变成静态舞台。
- **影响**：首轮只保留来访者、匪患传闻、商路异动、农庄求援、警戒状态变化，不做完整攻城与围城链路。

## 二、Agent 能力、Skill 与基模边界

### QA-07 游戏里的 `agent skill` 到底是什么？

- **问题**：这里说的 `agent skill` 到底指什么，为什么不能直接套传统游戏技能的理解。
- **澄清结论**：`agent skill` 是 Agent 在某类任务上的任务能力包，体现其方法、判断框架、流程经验、调用顺序和质量控制，不等同于传统 RPG 数值技能树。
- **影响**：TinyMB 后续验证的不是“点了哪个技能数值 +5”，而是“同样任务下，不同 skill 是否改变了执行路径、效率、稳定性和结果质量”。

### QA-08 `agent skill` 与角色数值/传统能力/权限有什么区别？

- **问题**：如果 Agent 的行动结果本来就受数值、传统能力和权限影响，那 `agent skill` 的独立价值在哪里。
- **澄清结论**：数值/传统能力负责门槛、公式、概率和硬性结果边界；权限负责能不能合法尝试；`agent skill` 负责“怎么想、怎么做、先做什么、如何回退”。它影响方法与路径，不直接改写数值结算公式。
- **影响**：能力模型被明确拆成三层：`基模` 影响推理质量，`agent skill` 影响任务方法，`角色状态/权限` 影响合法性与结算边界。

### QA-09 能否把游戏里的 `agent skill` 理解为“玩游戏的能力和技巧”？

- **问题**：是否可以把这类 skill 理解成 Agent 在游戏环境中的办事能力和玩法技巧。
- **澄清结论**：可以这样粗理解，但必须进一步落到“可复用、可配置、可作用于特定任务类型的能力包”这个层面，而不能停留在抽象熟练度描述。
- **影响**：后续设计 skill 时，需要从“哪些高频行为需要方法差异”出发，而不是从文学化技能名或传统职业树出发。

### QA-10 `agent skill` 存在必要性的前提是什么？

- **问题**：什么时候游戏环境中的 `agent skill` 才真的有存在必要，而不是多此一举。
- **澄清结论**：当前提是环境复杂、多路径、信息不完全、任务过程中经常需要取舍与回退，而裸基模难以稳定、可控、可解释地做出好结果时，skill 才有存在价值。
- **影响**：TinyMB 后续 skill 设计要围绕“高频失败点、白跑路径、能力缺口”来收敛，而不是先拍脑袋列一堆能力名。

### QA-11 为什么 MVP 首轮更适合固定一个基准模型？

- **问题**：如果 MVP 要验证 skill 价值，是否应该先固定一个能力没那么强的基准模型。
- **澄清结论**：是。主验证阶段先固定基准模型，可以减少变量，优先观察“有 / 无 skill”是否拉开行动过程与结果差异；跨模型比较保留为次验证。
- **影响**：`Q-10` 当前阶段结论被收敛为“先固定基准模型做主验证”，具体能力调整机制待 `Q-09` 收敛后再定稿。

### QA-12 “首版目录 / 挂点 / 原子能力”分别是什么意思？

- **问题**：这三个说法在 skill 设计里分别指什么。
- **澄清结论**：
  - `首版目录`：玩家最终能看见和选择的首版 skill 列表
  - `挂点`：skill 介入行为循环的位置，比如观察、拆解、路径选择、校验、回退
  - `原子能力`：可被多个 skill 复用的最小功能组件
- **影响**：后续 skill 设计会优先围绕“挂点”和“原子能力”抽象，而不是一开始就写满最终 skill 列表。

## 三、时间与行动结构

### QA-13 为什么“半天制 / 每天 2 行动”不合适？新时间模型要解决什么问题？

- **问题**：每天只有 2 次行动会让体验太像回合制，和 AIvilization 的连续行动感差太大，这个问题到底出在哪里。
- **澄清结论**：问题不在“移动要不要耗时”，而在旧口径把跨区移动和区域行动都压成了过粗的半天单位，导致跨区代价过高、节奏过硬。新模型改为：`连续时钟 + 10 分钟基本粒度 + 5~20 分钟跨区移动 + 同区连续办事`。
- **影响**：压力来源从“每天只有 2 次行动”改成“跨区折返成本 + 时段窗口 + 事件时效”，既保留规划压力，又避免强回合感。

## 四、接口与验证设计

### QA-14 什么是“结构化行动意图”？它由谁发给谁，在玩法里怎么起作用？

- **问题**：`结构化行动意图` 到底是什么，它在当前 TinyMB 架构中的发送关系和玩法作用是什么。
- **澄清结论**：它本质上是 Agent 向 `Server/Runtime` 提交的标准化动作申请单。玩家 Agent 和 NPC Agent 都在 `Act` 阶段输出它；`Runtime` 负责合法性、时间、资源、权限和结果结算。
- **影响**：后续 `P2`、`P3`、`P4` 都要围绕这个接口来展开：先定动作契约，再定观察包，再定日志 / 请示 / 能力依据展示。

### QA-15 `P2` 的“混合方案”到底是什么？

- **问题**：所谓混合方案，是否可以理解为“一个主行动 + 作为背景的 todo 计划与条件判断策略”。
- **澄清结论**：可以这样理解，但要更精确：它是“一个由 `Runtime` 权威执行的主动作 + 该动作之后的后续意图、条件判断、中断点与回退策略”。`Runtime` 只执行主动作，不把整段背景计划当作必须一次性结算的硬计划。
- **影响**：这套口径兼顾了 `Runtime` 的简洁校验和 Agent 规划差异的可见性，是 `P2` 当前推荐方向。

### QA-16 混合方案的优越性体现在哪些方面？

- **问题**：如果混合方案本质上只是“主动作 + 背景计划”，它相比纯原子方案或短计划方案到底强在哪里。
- **澄清结论**：它的优势在于同时避开两个极端：
  - 相比纯原子方案，它能保留 Agent 的规划能力、改道逻辑和能力缺口，便于比较基模 / skill 差异
  - 相比短计划方案，它不要求 `Runtime` 一次性承接整段计划，更适合动态世界、中途请示和状态变化
- **影响**：这让 TinyMB 后续更容易做路径对比、失败归因、请示插入、调试回放和 skill 价值验证。

### QA-17 混合方案的主要价值是“可追溯分析”，还是“提高 Agent 实际解决问题的能力”？

- **问题**：混合方案的主收益到底更偏验证，还是更偏实战提升。
- **澄清结论**：主收益是“可追溯、可分析、可验证”，次收益是“让 Agent 的办事过程更稳、更少白跑和误判”。它不会凭空提高模型上限，但会显著提高已有能力在动态游戏世界中的落地率、稳定性和可解释性。
- **影响**：TinyMB 当前阶段选择混合方案，首先是为了更好地验证 `基模 / Agent Skill` 差异，顺带提升实际办事稳定性。

### QA-18 当前动作边界由谁决定？

- **问题**：观察包里的 `available_actions` 是提示层，还是像真人界面里当前可点击的按钮那样，由游戏机制硬卡的合法动作集合。
- **澄清结论**：它应由 `Runtime` 权威裁定，等价于当前状态下 UI 可点击的合法动作集合。位置、时间、资源、权限、状态锁定等共同决定这轮能否提交某个 `primary_action`；模型可以在 `followup_plan` 里表达想做但当前不合法的下一步，但不能把不合法动作直接作为当前主动作。
- **影响**：`P2` 的动作契约与 `P3` 的观察包边界被明确切开：模型负责规划，`Runtime` 负责合法性。

### QA-19 `world_state_summary` 与 `memory_recall` 应该如何分工？

- **问题**：观察包里的世界摘要到底该放公共客观信息，还是放带有角色视角的“这对我意味着什么”；同时它和个人记忆召回的边界怎么切。
- **澄清结论**：`world_state_summary` 只放普通人普遍会知道的客观世界信息，如警戒升级、粮价上涨、求援将过期；`memory_recall` 才放角色过去记住、且这次被当前问题召回的个人记忆，如承诺、怀疑、未完成线索与关系记忆。当前观察与 `memory_recall` 共同作为后续决策输入。
- **影响**：客观世界态势与主观个人记忆被显式分开，后续 `P3`、`P4` 更容易保持日志、请示和回顾的口径清晰。

### QA-20 角色视角层该由脚本决定，还是由模型决定？

- **问题**：在 Route A 下，角色职业 / 派系 / 性格导致的“关注点差异”应由规则脚本来提取，还是直接交给大模型按角色设定自行判断。
- **澄清结论**：采用混合方案。设计师用规则定义角色默认关注权重与硬保底项，玩家通过长期方针轻调方向；模型 / skill 再结合当前问题做二次理解和少量例外提升。公共世界真相不因角色不同而变化，角色差异主要作用在记忆写入、召回优先级和最终筛选上。
- **影响**：这让 TinyMB 能同时保留角色味、模型灵活性与后续差异验证的可解释性。

## 五、计划与会话协同

### QA-21 这套计划系统建立前已经存在的 session 会发生什么？

- **问题**：在 `当前执行计划.md`、`活跃会话总表.md` 与 `子计划/` 建立之前，已经存在的历史 session 是否会被自动接管；如果不会，后续应如何处理。
- **澄清结论**：不会自动接管，也不会自动消失。若旧 session 的任务仍在推进，必须先人工补登记到 `活跃会话总表.md` 再继续；若状态不清，应先标为 `待人工核对` 并暂停可能冲突的任务；若实际已结束，只需把有效结论回写到长期文档，不必补登记为活跃 session。
- **影响**：`活跃会话总表.md` 记录的是“当前项目下所有仍活跃的 session”，而不是“当前窗口”；多窗口切换时先看该表，避免重复领取、并行冲突和历史会话状态漂移。

### QA-22 为什么《目录结构索引》之前没有随大量变更自动更新？

- **问题**：`TinyMB/目录结构索引.md` 在后续新增了 `当前执行计划.md`、`活跃会话总表.md`、`开发过程问答.md`、`子计划/` 等结构后，为什么没有像管理文档那样被自动同步更新。
- **澄清结论**：最初没有自动更新，是因为它作为根目录导航文档，不在当时的强制同步规则覆盖范围内。`fateclaw-background-doc.mdc` 主要约束 `MVP/Management/` 管理文档与 `Scope/`、`Design/` 的实质变更；`fateclaw-plan-system.mdc` 当时也只定义计划系统文件本身的职责与回写关系，没有把根目录的 `目录结构索引.md` 列为必须同步目标。现已补强：`fateclaw-doc-structure.mdc` 增加了“根目录导航索引”的维护触发，`fateclaw-plan-system.mdc` 也增加了“计划系统关键文件变化时同步 `目录结构索引.md`”的专项触发。
- **影响**：以后目录结构、管理职责、计划系统关键文件或快速入口发生变化时，`目录结构索引.md` 应进入自动同步链路，不再只是“有维护价值但没人管”的导航文档。

### QA-23 文档类工作和代码类工作应该默认在哪个目录执行？这条约定是否必须先升成项目级 rules？

- **问题**：为了避免再次出现“实际改动落在 `worktree`，但主开发目录看不到”的混乱，TinyMB 后续应如何约定文档类工作与代码类工作的默认执行目录；这条约定是否必须先写成项目级 `.cursor/rules` 才能正常运行。
- **澄清结论**：当前采用最小可执行约定（v1）：`设计文档 / 管理文档 / 计划文档` 默认在主开发目录 `D:/01-工作/Garena/GI/FateClaw` 执行；`代码实现 / 并行开发 / 高风险改动` 才进入项目内 `/.worktrees/`。只要该约定已写入 TinyMB 的长期文档，并被当作开工前 / 收尾前检查执行，它就可以先正常运行；项目级 `.cursor/rules` 只是在需要跨新会话、跨项目、尽早生效时再补的增强项，不是本轮落地前置条件。
- **影响**：后续 session 在开始工作时，先判断任务类型再决定工作目录；若进入 `worktree`，必须在收尾前显式选择“继续留在 `worktree` / 回灌主开发目录 / 通过 git 集成”。这样即使暂时不补项目级 rules，也能先把流程稳定跑起来。

## 六、日志 / 请示与可观察性接口

### QA-24 `P4` 的日志 / 请示 / 状态回顾格式应该怎么收口？

- **问题**：`P4` 的日志、请示包和状态回顾，到底应该分别设计三套格式，还是共用一套底层结构；同时它应优先偏玩家可读性还是调试可追踪性，以及 `MVP-Must` 必填字段是否需要逐一解释作用。
- **澄清结论**：采用“**双层结构**”收口：底层统一使用 `decision_record`，对外再派生第一人称的 `log_card / ask_card / review_card`。玩家可读层优先保证顺滑的第一人称汇报，底层结构化字段优先保证回放、排错、失败归因和后续基模 / skill 差异验证。玩家 Agent 与命名 NPC 共用同一套核心 schema，只在展示权限与可见性上裁剪；`MVP-Must` 必填字段需要逐一写明开发用途，不能只列字段名。
- **影响**：`P4` 之后，TinyMB 的可观察性接口不再是零散的“日记文案要求”，而是统一的记录底座：实现阶段可以按字段职责拆模型输出、Runtime 回传和展示层映射；后续做 skill 配置展示、能力依据解释和状态回顾聚合时，也不需要再发明第二套数据结构。
