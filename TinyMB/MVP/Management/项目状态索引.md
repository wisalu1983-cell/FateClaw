## 一、项目背景

### 1.1 项目名称与定位

TinyMB（Tiny Mount & Blade）—— 骑马与砍杀极简版

TinyMB 是 **AIvilization Plus** 的 MVP 版本，以骑马与砍杀的核心玩法为基底，极简化后接入多 Agent 框架，用资源、关系和派系取舍来强化原始 AI 沙盒体验的目标感与博弈张力。

### 1.2 来龙去脉

#### 前序项目：FateClaw

FateClaw 最初计划做一个 Agent 驱动的圣杯战争模拟游戏（详见 `Brainstorming/Agent圣杯战争提案-对话提炼.md`）。MVP v1 阶段完成了 4 份 A 级设计文档（核心循环、数值框架、行动系统、情报系统），但复杂度严重膨胀，暂停并封存（详见 `FateAgent/MVP/Management/MVP-v1设计讨论封存总结.md`）。

#### 核心玩法选型

暂停后进行了系统化的核心玩法对标脑暴（详见 `Brainstorming/玩法对标与脑暴/02-核心玩法对标脑暴.md`），评估了 8 个候选方案，结论是 **K. 极简骑砍** 综合最强——骑砍本身就是"角色在持续运转的沙盒世界中闯荡"，与 AIvilization 架构同构，且是唯一在全部 10 种 Agent 能力模式上均达到高度利用、不依赖任何未成熟能力的方案。

#### TinyMB 的目标

基于上述结论，TinyMB 的目标是：

1. 将骑砍核心玩法极简化为可被 Agent 框架驱动的规则集
2. 基于该规则集开发可运行的 MVP
3. 验证核心体验假设和 Trumen 多 Agent 框架的技术可行性

### 1.3 技术底座

- **Trumen 多 Agent 框架**：Client / Agent Framework / Server(Runtime) 三层结构
- **Agent 框架职责**：驱动 Observe → Think → Decide → Act 循环
- **与玩法解耦**：框架通过标准化接口接收具体游戏的环境适配模型（身份/环境/行为/决策约束四类定义）
- **部署拓扑**：Agent Framework 按角色类型拆分部署——NPC Agent 在服务端运行，玩家 Agent 在用户侧运行（用户选择基模）。两类 Agent 对 Runtime 的接口相同：接收观察包、输出结构化行动意图、由 Runtime 统一校验和结算
- **行为框架约束**：玩家 Agent 的行为框架由官方系统 prompt 锁定（不可修改），用户选择基模只影响推理质量
- **等价模型**：用户侧玩家 Agent = 游戏环境中的 OpenClaw 式 Agent Runtime（官方系统 prompt + 用户基模 + 游戏行动工具集）

详见 `Brainstorming/多Agent框架相关/多Agent框架讨论总结.md`

### 1.4 核心体验假设

我相信，喜欢 AI 沙盒与轻策略体验的玩家，在通过对话引导一个有自主意志的 Agent 于持续运转的城邦中闯荡时，会因为简化版《骑马与砍杀》规则带来的资源、关系与派系取舍，以及玩家意图与 Agent 办事能力（基模、agent skill、角色数值/传统能力、权限）共同塑造行动过程与结果，而明确感知不同基模与 Agent Skill 组合会在同等任务条件下产生不同的执行规划、路径、效率与结果，并愿意据此继续调整指令与迭代 Agent Skill。

**当前验证信号**：

- 玩家会主动多次查看 Agent 日志、NPC 动向和世界变化。
- 玩家会主动介入至少一次资源、关系或派系取舍，并愿意承担由此带来的后果。
- 玩家能复述至少一条由 NPC 自主行动和玩家介入共同产生的连锁事件，并表示想继续观察后续发展。
- 玩家能明确感知不同基模与 Agent Skill 组合在同等任务条件下会产生不同执行规划、路径、效率与结果，并据此调整指令、迭代 Agent Skill。

### 1.5 设计约束

| 约束 | 来源 |
|---|---|
| Agent 是一个"人"（有日常行为、情绪、日记） | AIvilization 架构 |
| 世界连续运行（非回合制） | AIvilization 架构 |
| 玩家引导而非控制 | AIvilization 架构 |
| 多 Agent 自主交互 | AIvilization 架构 |
| 放置/挂机体验 | AIvilization 架构 |
| 空间/地图维度 | AIvilization 架构 |
| Agent 内心可观察 | AIvilization 架构 |
| 依赖已成熟 Agent 能力，避开未成熟能力 | Agent 技术边界 |
| 开放式行动结果由玩家目标、基模、agent skill、角色状态与权限共同影响 | OpenClaw 式 Agent 设计 |
| agent skill 是任务能力包，不等同于传统数值技能树 | OpenClaw 式 Agent 设计 |
| 玩家 Agent 的四步循环效率与质量必须显著受基模和 agent skill 影响 | OpenClaw 式 Agent 设计 |
| 先回答"最小系统是什么" | MVP v1 复杂度膨胀教训 |

## 四、阶段性结论

### 结论 1：骑砍与 AIvilization 架构的同构性已验证

骑砍的核心循环（目标驱动的连续行动流、多领主自主行为、势力兴衰演化）与 AIvilization 架构的 7 项约束天然兼容。调研未发现结构性冲突。

### 结论 2：骑砍策略层的核心是四变量平衡

Money / Renown / Honor / Relationships 四个变量相互牵制，构成骑砍策略层的灵魂。极简化时数值骨架至少需要覆盖这四个维度。

### 结论 3：极简化方向已明确

MVP 不需要还原骑砍的完整规模（22~57 城镇），只需保留机制的功能分化逻辑。

### 结论 4：城堡可与城镇合并（行动-聚落映射发现）

通过系统梳理骑砍全部行动与聚落层级的对应关系，发现：
- **城镇独占 8 项行动**（交易、同伴、佣兵、工坊、竞技、战俘、行会任务、酒馆），是行动最丰富的聚落类型
- **城堡没有任何完全独占的行动**——城堡能做的一切，城镇内部的城堡区都能做。城堡的独特价值纯粹在军事/战略层面
- **村庄仅独占 1 个核心行动**（招募基础兵源），可不作为独立地图节点而作为城镇的附属属性

因此建议世界结构从"城镇/城堡/村庄"三类简化为"城镇 + 村庄/哨卡"两类，城堡功能合并入城镇。详见 `参考资料/骑砍行动与聚落层级映射.md`

### 结论 5：单城镇方案可行（核心机制验证）

对骑砍 30 项核心玩法机制逐条验证后确认：
- 19 项在单城镇中完整保留，10 项变形保留（核心逻辑不变，空间尺度缩小），仅 1 项丧失（城际套利，非核心）
- 限制的是玩家行为自由度（空间选择减少、角色身份受限、进阶天花板降低）和战略纵深（从广度转为深度），而非核心机制
- 最终采用单城镇方案：石冠城，母稿 6 个功能区域 + 3 大派系；首轮可验证原型启用 4 个活跃区域

详见 `参考资料/单城镇方案核心机制验证.md`
