# TinyMB MVP 范围文档

> 版本：v0.6
> 日期：2026-04-02
> 状态：已确认（已补充 Agent skill 概念边界；P1 时间结构与行动粒度、P2 结构化行动意图格式已定稿，具体 skill 集待收敛）

---

## 1. 体验假设

**核心假设**：

我相信，喜欢 AI 沙盒与轻策略体验的玩家，在通过对话引导一个有自主意志的 Agent 于持续运转的城邦中闯荡时，会因为简化版《骑马与砍杀》规则带来的资源、关系与派系取舍，以及玩家意图与 Agent 办事能力（基模、agent skill、角色数值/传统能力、权限）共同塑造行动过程与结果，而明确感知不同基模与 Agent Skill 组合会在同等任务条件下产生不同的执行规划、路径、效率与结果，并愿意据此继续调整指令与迭代 Agent Skill。

**验证标准**：

- 玩家会主动多次查看 Agent 日志、NPC 动向和世界变化，而不是只看一次结果后退出。
- 玩家会主动介入至少一次资源、关系或派系取舍，并愿意承担由此带来的后果。
- 玩家能清晰复述至少一条由 NPC 自主行动和玩家介入共同产生的连锁事件，并表示想继续观察后续发展。
- 玩家能明确感知不同基模与 Agent Skill 组合在同等任务条件下会产生不同执行规划、路径、效率与结果，并据此调整指令、迭代 Agent Skill。

## 2. 核心循环

### 微循环（10-30秒）

**输入**：玩家阅读当前状态，并通过自然语言给出一个短期目标或行动指令。
**处理**：Agent 基于玩家目标、自身基模、agent skill、角色状态（数值/传统能力/权限）、位置、资源、关系和近期事件判断行动方案，并以**结构化行动意图（混合方案）**输出：一个权威主动作 + 后续意图 / 中断点 / 回退策略，交由 Runtime 校验和结算。
**反馈**：系统返回行动结果、日志更新、资源和关系变化，以及新机会或新风险。
**驱动**：玩家因看见世界被自己"轻推一下就发生变化"而想继续观察与介入。

### 中循环（3-10分钟）

**目标**：围绕一个短期目标推进，例如赚钱、站队、争取准入、化解威胁。
**过程**：在 2-3 个区域间移动，接触 NPC 或事件，执行任务或做取舍，并承受后果。
**奖励**：获得金钱、关系、声望、后续任务机会或新的局势变化。
**变化**：不同的派系立场、NPC 状态和事件组合，会让下一轮中循环出现不同走向。

### 宏循环（单局/单天）— MVP 中为占位实现

**结构**：玩家在一段持续运转的城邦生活中，观察几天到一周内的人物关系和局势变化。
**MVP 处理**：先以 7-14 天游戏内时间的短周期场景验证"世界是否活起来"。长期进阶（从外来者到城主的完整弧线）、完整城主循环和大规模政治演化暂不实现——只要玩家能感受到"世界在演变、我的介入有后果"就够了。

## 3. 功能清单（MoSCoW）

### 术语说明

- **命名 NPC**：有唯一名字、性格、个人目标和派系身份的完整 Agent 实体，跑完整的 Observe→Think→Decide→Act 循环，由服务端 LLM 驱动决策。命名 NPC 是"世界活着"这一体验的核心载体。
- **非命名角色**：没有独立名字和个性的背景存在（如"市集摊贩"、"巡逻卫兵"），不跑 Agent 循环，仅作为环境描述或规则触发器的一部分。MVP 中非命名角色只需要以文本形式存在。
- **基模**：驱动玩家 Agent 核心理解、推理、规划与表达能力的基础模型，运行在用户侧。基模不等同于具体 agent skill，但会显著影响同类任务下的判断深度、稳定性与执行质量。用户选择基模只影响推理质量，角色的行为框架由官方系统 prompt 锁定。
- **Agent skill**：指 Agent 在游戏环境中处理某类事务的任务能力包，体现其方法、判断框架、流程经验和办事技巧，会影响同一目标下的行动方案、执行效率与结果质量。这里的 agent skill 不等同于传统 RPG 技能树。
- **角色数值/传统能力**：指战力、体力、声望、熟练度等可被规则直接引用或结算的角色属性，用于门槛、概率或公式计算。
- **权限**：指 Agent 当前拥有的身份、准入资格、可访问区域、可调用资源与可支配对象范围，决定哪些行动在当前世界状态下可被合法尝试。
- **实现复杂度**：标注为低/中/高，反映 AI coding 所需的迭代轮次和调试难度，而非人天。

### Must Have — 删了核心循环就跑不起来

| #  | 功能                        | 说明                                                                                                                                                                      | 为什么是 Must                                                                                     | 复杂度 |
| -- | --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- | ------ |
| M1 | 单城镇持续运转世界骨架      | 石冠城时间推进（连续时钟 + 10 分钟基本粒度）、区域状态、基础事件刷新、每 3 天结算一次资源消耗                                                                             | 没有持续世界，AIvilization 的基本体验不成立                                                       | 中     |
| M2 | 玩家对话引导 Agent          | 玩家用自然语言给出目标或指令，Agent 基于自身基模、agent skill、角色状态与权限规划执行，并以结构化行动意图向 Runtime 提交主动作后回报结果                                       | 没有引导机制，核心交互会退化成纯旁观，也无法验证"玩家意图 × 基模 × Agent 办事能力"               | 中     |
| M3 | 6 名命名 NPC 的自主循环     | 6 名命名 NPC 基于性格、目标和状态在城中移动、行动和互动；其余角色以文本暗示存在                                                                                             | 没有自主他者，就没有"世界活着"的感受                                                              | 高     |

| M5 | 四变量基础版 + 资源消耗压力 | Money（金钱）/ Renown（声望，影响准入门槛）/ Honor（荣誉，影响 NPC 态度）/ Relationships（派系关系 + 个人关系）全保留最简实现；随从有日常消耗（粮食或金钱），不维护会流失 | 四变量是骑砍策略层的灵魂——缺 Renown 则无进阶门槛，缺 Honor 则无道德抉择，缺资源消耗则无取舍压力 | 中     |
| M6 | 基础任务/事件与站队后果     | 提供短期目标，并让不同立场产生不同结果                                                                                                                                    | 没有后果链，博弈张力无法被验证                                                                    | 中     |
| M7 | 极简风险/冲突结算           | 用最简规则处理失败、受伤、损失和收益                                                                                                                                      | 没有风险，选择不会变得有意义                                                                      | 低     |
| M8 | 行动日志与世界状态回看      | 让玩家看见 Agent 与 NPC 的行为轨迹和变化                                                                                                                                  | 没有可观察性，体验假设无法验证                                                                    | 低     |

## 4. 技术约束

- 平台：开发态原型，优先本地 PC 调试环境
- 引擎/技术栈：沿用 Trumen 多 Agent 框架，并补最小游戏环境适配层
- 资产策略：以文本、日志和最简可读 UI 为主，必要时使用占位图形，不做发售级包装
- 实现方式：AI coding，通过迭代式 prompt 驱动开发
- 能力口径：开放式行动结果由玩家目标、基模、agent skill、角色状态与权限共同影响；首版 agent skill 集在阶段 2 收敛
- 核心验证点：玩家 Agent 必须在游戏环境中完整跑通 Observe→Think→Decide→Act 基本循环；所用基模与 agent skill 要显著影响循环效率与质量，并且这种差异对玩家可感知
- 部署架构：Agent Framework 按角色类型拆分——命名 NPC 的决策 LLM 在服务端运行；玩家 Agent 的推理 LLM 在用户侧运行（用户选择基模）。两类 Agent 对 Runtime 的接口相同
- 行动输出契约：所有 Agent（不论部署位置）输出的都是**混合型结构化行动意图**；`Runtime` 只权威执行一个主动作，后续意图 / 中断点 / 能力依据 / 回退用于请示、日志、回放和差异验证
- 行为框架约束：玩家 Agent 的行为框架由官方系统 prompt 锁定，用户仅选择基模（影响推理质量，不影响角色身份与规则框架）

## 6. 开放问题

| #  | 问题                                                             | 影响范围                               | 需在哪个阶段解决 |
| -- | ---------------------------------------------------------------- | -------------------------------------- | ---------------- |
| Q1 | 战斗结算采用什么极简模型？ | 影响战斗风险、自动结算稳定性和数值压力 | 阶段 2/3 |
