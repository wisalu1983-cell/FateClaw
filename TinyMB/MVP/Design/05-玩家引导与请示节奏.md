# 玩家引导与请示节奏

> **所属**：极简骑砍规则集 · 模块 5
>
> **设计目标**：定义玩家如何与自己的 Agent 互动——不是直接操控，而是通过对话引导。同时定义 Agent 在什么场景下自主行动、什么场景下请示玩家、以及玩家离线时的行为策略。
>
> **核心约束**：玩家引导而非控制（AIvilization 架构约束）。Agent 是一个有自主判断力的"人"，玩家更像是 Agent 的"军师/顾问"而非"遥控器"。

---

## 0. MVP 回标说明

本文档同时保留两层口径：

- **母稿**：完整四级自主等级、复杂请示节奏与长期方针的完整设计。
- **首轮可验证原型回标稿**：只保留验证当前体验假设所需的最小交互层次。

### 首轮可验证原型（MVP-Must）

- 玩家 Agent 采用**默认自主 + 重要决策请示 + 未回复安全回退**的最小基线
- 玩家 Agent 的**基模必须支持通过配置参数切换**
- Agent Skill 采用**预制库 + 用 / 不用 + 槽位限制**的方式接入
- 日记 / 请示 / 状态回顾里必须能让玩家感知**当前基模、已启用 Agent Skill 与本次行动能力依据**
- 日志 / 请示 / 状态回顾的底层记录采用统一的 `decision_record`，对外再派生 `log_card`、`ask_card`、`review_card`

### 首轮原型后放（Later）

- 完整四级自主等级
- 更细的请示频率调参
- 更复杂的长期方针与积压机制

## 1. 玩家与 Agent 的交互模式

| 模式 | 说明 | 何时发生 |
|---|---|---|
| **对话指令** | 玩家通过自然语言告诉 Agent 想做什么（对话与 Agent 推理在用户侧实例中运行） | 玩家在线时随时可发起 |
| **请示响应** | Agent 遇到重要决策，暂停并询问玩家意见 | Agent 触发请示条件时 |
| **长期方针** | 玩家设定优先级和行为偏好，Agent 据此自主行动 | 玩家在线时设定，离线时生效 |
| **回顾观察** | 玩家查看 Agent 的日记、行动记录、世界变化 | 玩家上线时或随时查看 |

## 2. 首轮可验证原型的请示基线（MVP-Must）

### 2.1 默认自主

不命中重要决策条件时，Agent 默认自主执行，并在行动后通过日志回报结果。

| 行动类型 | 典型例子 | 处理方式 |
|---|---|---|
| 日常移动与补给 | 去市集采购、去农庄买粮、区域间移动 | 直接执行，事后记录 |
| 常规信息搜集 | 酒馆打听消息、观察局势、与普通角色闲聊 | 直接执行，事后记录 |
| 低风险任务推进 | 一般跑腿、基础交易、小额委托 | 直接执行，事后记录 |
| 小幅关系 / 资源波动 | +1/-1 级别关系变化、小额收支 | 直接执行，避免信息过载 |

### 2.2 重要决策请示

命中以下条件时，Agent 必须生成请示包，说明建议、风险、替代路径和能力依据。

| 触发条件 | 典型例子 | 为什么要请示 |
|---|---|---|
| 明确站队或改变立场 | 接受涉及派系站队的重要任务、加入 / 退出派系 | 会改变后续关系链和长期路线 |
| 高风险高后果行动 | 对命名 NPC 发起冲突、接受高风险任务 | 失败成本高，不应默认自动执行 |
| 大额资源投入 | 大额交易、重要投资、关键资源押注 | 会影响后续几天的行动空间 |
| 发现能力缺口 | 当前基模 / 已启用 Agent Skill 不适配目标，Agent 建议改用别的路线或调整 skill | 这是首轮原型要验证的核心差异点 |
| 关键队伍变化 | 招募关键同伴、替换关键配置 | 会长期影响行动方式与结果 |

### 2.3 未回复安全回退

玩家未及时回复时，Agent 按以下安全规则处理：

| 情况 | 回退规则 |
|---|---|
| 可逆、低风险 | 暂缓或按最保守方案执行 |
| 可逆、时间敏感 | 按 Agent 建议的稳妥默认值执行，并在日志中明确说明 |
| 不可逆、重大后果 | 不执行，继续等待玩家指令 |

### 2.4 与结构化行动意图的关系（MVP-Must）

首轮原型中，请示机制与 `P2` 的结构化行动意图直接绑定：

- 命中请示条件时，Agent **不应直接提交高风险 `act` 主动作**
- 当前轮应改为提交 `ask` 型主动作，由 `Runtime` 进入请示 / 等待状态
- 请示包至少复用以下字段：
  - 拟执行的 `primary_action`
  - `followup_plan`
  - `interrupt_points`
  - `capability_basis`
  - `capability_gap`
  - `fallback`

这样做的目的，是让玩家看到的不只是“要不要同意”，而是：

- Agent 原本打算先做什么
- 后续原本准备怎么接
- 为什么这里必须请示
- 如果不同意，Agent 会怎样回退

### 2.5 P4：日志 / 请示包 / 状态回顾的统一记录底座（MVP-Must）

首轮原型中，`P4` 不把“日志”“请示”“状态回顾”当成三套彼此独立的输出格式，而是采用**双层结构**：

- **底层记录层**：统一的 `decision_record`
- **玩家展示层**：第一人称可读的 `log_card`、`ask_card`、`review_card`

这样做的目标有三点：

- 让玩家读到的是自然、顺滑、像 Agent 自己在汇报的内容
- 让系统内部保留稳定的结构化字段，便于回放、调试、失败归因与后续基模 / Agent Skill 差异验证
- 让玩家 Agent 与命名 NPC 共用同一套记录底座，只在展示权限和可见性上做裁剪

#### `decision_record` 的最小分块

| 区块 | 作用 | 首轮最小内容 |
|---|---|---|
| `record_meta` | 标识这条记录是谁、何时、因什么产生的 | `record_id`、`actor_id`、`actor_type`、`timestamp`、`source_type`、`related_intent_id` |
| `situation` | 交代触发这条记录时的情境 | `location_id`、`current_goal`、`trigger_event` |
| `decision` | 说明 Agent 原本要做什么、为什么停在这里 | `primary_action_snapshot`、`followup_plan_summary`、`interrupt_reason` |
| `capability` | 暴露这次路线选择背后的能力依据与缺口 | `basis`、`gap`、`why_this_route` |
| `outcome` | 说明当前是已完成、待决还是被阻断，以及已发生的结果 | `status`、`result_summary`、`costs_or_changes` |
| `fallback` | 给出条件不足或玩家未回复时的安全默认方案 | `fallback_plan` |
| `next_step` | 为下一轮循环、后续日志和状态回顾提供衔接 | `proposed_next_action`、`awaiting_player_input` |

`decision_record` 的设计边界应锁定为：

- 它**复用** `P2` 中的 `primary_action / followup_plan / interrupt_points / capability_basis / capability_gap / fallback`
- 它**不替代** `P2` 的行动契约，也不新增第二套权威执行格式
- 它负责把“这轮发生了什么、为什么这么做、当前卡在哪、接下来准备怎么做”整理成统一记录底座

#### 最小结构示意

  },

#### `MVP-Must` 必填字段与作用说明

下表中的字段不只是在文档里“列名备案”，而是要明确它们在后续开发中的职责，避免实现阶段再重新猜测字段用途。

| 字段 | 是否必填 | 作用（面向后续开发） |
|---|---|---|
| `record_meta.record_id` | 是 | 这条记录的稳定主键。后续做日志列表、请示详情、状态回顾聚合、回放定位时，都用它来唯一引用，避免展示层和调试层各自生成不同 ID。 |
| `record_meta.actor_id` | 是 | 标明是哪一个 Agent 产出的记录。没有它，玩家 Agent 与命名 NPC 的记录无法共存到统一流水里，也无法按角色筛选历史。 |
| `record_meta.actor_type` | 是 | 决定这条记录走玩家展示逻辑还是 NPC 可见性逻辑。后续做“哪些字段默认可见、哪些字段需要关系/情报解锁”时，需要先分清 `player / npc`。 |
| `record_meta.timestamp` | 是 | 统一时间排序依据。后续的时间线展示、阶段汇总、跨区行动回看都要靠它稳定排序，不能只靠生成先后。 |
| `record_meta.source_type` | 是 | 告诉系统这条记录当前主要服务于 `log / ask / review` 哪种展示入口。它是展示层路由开关，避免同一条记录被错误渲染成不合适的卡片。 |
| `record_meta.related_intent_id` | 是 | 把 `P4` 记录回链到 `P2` 的结构化行动意图。后续如果要做“这条日志对应哪次 intent”“这次请示源自哪条 `ask` 主动作”的排查，就靠这个字段。 |
| `situation.location_id` | 是 | 交代事情发生在哪个区域。后续状态回顾、区域筛选、地图高亮和“同一区域连续办事”回放都要引用它。 |
| `situation.current_goal` | 是 | 把当前记录锚定到一个清晰目标上。否则多条日志会只剩散碎行动，看不出“这轮在推进什么事”。 |
| `situation.trigger_event` | 是 | 说明这条记录为什么此刻生成，是“动作刚结算”“命中请示条件”还是“进入阶段回顾”。后续做 ask 触发分析和 review 聚合时都需要这个触发点。 |
| `decision.primary_action_snapshot` | 是 | 记录当时准备执行或刚执行的主动作快照。之所以保留快照而不是只回查 intent，是为了防止后续 intent 被改写或上下文丢失后，日志失去可读性。 |
| `decision.followup_plan_summary` | 是 | 承接 `P2` 的后续意图，但压成可展示的摘要。后续 ask 卡和状态回顾都要告诉玩家“你如果不干预，我本来准备怎么接着做”。 |
| `decision.interrupt_reason` | 是 | 解释为什么这轮没有继续往下执行，是因为高风险、不可逆、站队压力还是能力缺口。后续请示包正文、调试统计和 ask 触发归因都依赖这个字段。 |
| `capability.basis` | 是 | 列出本次行动依赖了哪些能力依据，如基模、已启用 skill、数值、权限、关系。它是后续能力徽章、差异对比和解释型日志的核心数据来源。 |
| `capability.gap` | 是 | 列出当前目标上明显存在但尚未补齐的能力缺口。后续要验证 skill 是否减少白跑、是否改变路线，首先就要让缺口能被稳定记录下来。 |
| `capability.why_this_route` | 是 | 用一句话把“能力依据 / 能力缺口”与“为什么选这条路”连起来。没有它，玩家只能看到原始字段堆叠，很难直接理解 Agent 的判断逻辑。 |
| `outcome.status` | 是 | 标识这条记录当前是 `done / pending / blocked / fallback` 中哪一类。它直接决定这条记录被渲染成日志、请示还是待处理状态。 |
| `outcome.result_summary` | 是 | 对当前结果做一句话收口。后续列表页、状态回顾摘要和通知提醒不可能总展开长正文，必须有稳定的短摘要字段。 |
| `outcome.costs_or_changes` | 是 | 记录这次动作带来的时间、资源、关系、风险或世界状态变化。后续 review 聚合、变化对比和“这轮到底付出了什么代价”都靠它汇总。 |
| `fallback.fallback_plan` | 是 | 统一记录“如果你不批 / 条件不足 / 目标缺席，我会怎么退”。这是安全回退机制的落地点，后续未回复自动处理和失败归因都要读取它。 |
| `next_step.proposed_next_action` | 是 | 指向下一步最合理的衔接动作。后续让 Agent 连续办事、让玩家快速理解当前上下文，都需要一个稳定的 next step 字段。 |
| `next_step.awaiting_player_input` | 是 | 显式标记当前是否卡在玩家决策上。后续请示堆积合并、待办列表和通知逻辑都不能靠猜，要靠这个字段明确判断。 |

以下字段首轮可保留为**非必填扩展位**：

- `situation.relevant_entities`
- `outcome.risk_note`
- `fallback.fallback_trigger`
- `next_step.open_question`

它们有价值，但不是首轮把 `P4` 跑通所必需的最小闭环。

#### 三种展示对象的映射关系

| 展示对象 | 触发时机 | 主要取数来源 | 玩家看到的重点 |
|---|---|---|---|
| `log_card` | 动作已结算、或回退已执行 | `record_meta + situation + decision + capability + outcome + next_step` | 我做了什么、为什么这么做、现在看到什么、接下来准备做什么 |
| `ask_card` | 命中请示条件，当前进入等待决策 | `situation + decision + capability + fallback + next_step` | 我卡在哪、原计划是什么、为什么来问你、我建议怎么走、你不回我会怎么退 |
| `review_card` | 玩家主动查看近况、阶段结束、多个 ask 积压时 | 聚合多条 `decision_record` 的 `outcome + next_step` | 这段时间推进了什么、发生了哪些关键变化、现在有哪些待决事项 |

三者的边界也应锁定：

- `log_card` 与 `ask_card` 都来自单条 `decision_record`
- `review_card` 只做聚合，不额外发明新事实
- 同一条 `decision_record` 在同一时刻只主派生一种主展示对象，避免一份事实被多次重复叙述

#### 玩家 Agent 与命名 NPC 的共用 / 裁剪规则

- 玩家 Agent 与命名 NPC **共用同一套 `decision_record` 核心 schema**
- 玩家 Agent 默认完整使用 `log_card / ask_card / review_card`
- 命名 NPC 默认只公开 `log_card / review_card`
- 命名 NPC 即使底层记录了完整 `capability` 与判断过程，对玩家是否可见仍受关系值和情报手段控制
- 未来若要做“通过情报揭露 NPC 真实想法”，优先开展示层可见性，不重新发明第二套底层字段

## 3. 完整四级自主等级（Later 母稿）

以下四级方案保留为后续扩展母稿，不作为首轮可验证原型的 `MVP-Must` 实现要求。

### Level 0：完全自主（不通知）

Agent 自行处理，玩家事后通过日记得知。

| 行动 | 理由 |
|---|---|
| 日常移动（去市集买粮、回住所休息） | 生存性日常，无需决策 |
| 与普通NPC的日常对话 | 常规社交，影响极小 |
| 轻微的关系变化（+1/-1） | 波动太小，通知会造成信息过载 |

### Level 1：自主执行，事后通知

Agent 执行后在日记中记录，玩家下次查看时可以看到。

| 行动 | 理由 |
|---|---|
| 完成日常任务（巡逻、运货等） | 例行公事，结果有意义但决策简单 |
| 参加比武（非重大赛事） | 有风险但可接受 |
| 随从日常训练 | 长期积累型行为 |
| 应对小规模匪帮遭遇 | 被动触发、需要立即反应 |

### Level 2：请示推荐（Agent 给出建议，等待确认或修改）

Agent 暂停行动，展示当前情况和建议方案。玩家未响应时，Agent 按建议方案执行。

| 行动 | 理由 |
|---|---|
| 接受/拒绝重要任务 | 可能涉及派系站队或较大风险 |
| 参加派系议事并投票 | 政治选择影响深远 |
| 接受 NPC 的宴会/密谈邀请 | 可能改变关系网 |
| 投资工坊/大额交易 | 重要经济决策 |
| 招募新同伴 | 长期影响队伍构成 |

### Level 3：请示必须（Agent 暂停，必须等待玩家决策）

不可逆行动，Agent 不会在无玩家指示的情况下执行。

| 行动 | 理由 |
|---|---|
| 加入/退出派系 | 改变角色核心身份 |
| 背叛当前派系 | 不可逆的关系破坏 |
| 与命名 NPC 开战 | 可能导致重要角色死亡/被俘 |
| 竞选派系领袖/城主 | 角色发展的关键节点 |
| 释放/处决重要囚犯 | 荣誉和关系的重大转折 |

## 4. 请示节奏设计（MVP-Must）

| 场景 | 请示频率 | 说明 |
|---|---|---|
| 玩家在线、积极互动 | 每完成 20~40 分钟的连续办事，或命中重要节点时 1 次请示 | 玩家感受到参与感，同时保留连续行动感 |
| 玩家在线、挂机观察 | 每完成 40~90 分钟的连续办事，或命中关键节点时 1 次请示 | 只在重要节点打扰，避免信息轰炸 |
| 玩家离线 | 0 次请示 | Agent 按长期方针自主行动 |

首轮原型不再按“每天 2 个行动机会”设计请示节奏，而改为按**连续办事时长 + 关键节点触发**来决定何时打断玩家。

### 请示积压机制

多个请示事件同时触发时，合并为一个"近况汇报+待决清单"：

```
【Agent 近况汇报】
今天 09:10 我去了酒馆区，打听到商盟正在囤积粮食（获得新情报）。
09:40 我原本准备直接去城主堡交涉，但当前没有启用“交涉”类 Agent Skill，直接出面风险偏高。

【待决事项】
1. 待执行主动作：`ask -> 是否改为直接交涉`
2. 当前后续打算：先去农庄确认平民公社态度，再决定是否去城主堡
3. 能力缺口：缺少更强的交涉能力；若你不批准，我会按保守路线继续搜集证据
```

## 5. 长期方针系统

| 方针维度 | 选项 | 影响 |
|---|---|---|
| **经济优先级** | 攒钱 / 投资 / 不关心 | Agent 是否主动寻找赚钱机会 |
| **派系倾向** | 偏向某派系 / 保持中立 / 见机行事 | 影响任务接受和社交选择 |
| **军事态度** | 积极扩军 / 维持现状 / 精兵路线 | 影响招募和训练行为 |
| **社交风格** | 广泛交友 / 重点深耕 / 低调行事 | 影响社交行为频率和对象选择 |
| **风险偏好** | 保守 / 平衡 / 激进 | 影响是否接受高风险高回报的机会 |

首轮原型保留：**经济优先级 / 派系倾向 / 风险偏好**

`Later`：军事态度、社交风格等更细方针维度

玩家离线时，Agent 按方针自主行动，所有行动降级为 Level 0 或 Level 1。

## 6. Agent 日记、请示卡与状态回顾（MVP-Must）

### 日记内容

首轮原型中，底层先生成 `decision_record`，玩家最常直接看到的是**第一人称 `log_card`**。每段连续行动结束后，或在关键节点被请示打断时，默认生成如下可读条目：

| 组成部分 | 内容 | 示例 |
|---|---|---|
| **行动记录** | 做了什么、在哪里、和谁 | "今天 09:10 在酒馆区和老酒·加利聊了最近的粮价" |
| **内心想法** | 为什么做出这个选择 | "我现在没有启用更强的交涉能力，先搜集情报更稳妥" |
| **观察与感受** | 对事件和NPC的主观看法 | "杜兰今天明显在回避农庄的话题，这不像偶然" |
| **能力依据** | 本次行动主要依赖的基模 / Agent Skill / 数值 / 权限 | "当前启用了调查类 Agent Skill，所以我优先选择酒馆打听而不是直接去城主堡" |
| **对未来的打算** | 接下来想做什么 | "如果你同意，我明天想先去农庄确认消息，再决定是否交涉" |

首轮原型要求日志 / 请示中可见以下信息：

- 当前玩家 Agent 的基模
- 当前已启用的 Agent Skill
- 本轮 `primary_action` 与后续打算摘要
- 本次行动的能力依据或能力缺口
- 若能力配置不适配当前目标，Agent 给出的替代路线

### 请示卡（`ask_card`）

命中请示条件时，玩家看到的不是裸字段，而是一张第一人称请示卡。它至少应包含：

- **一句话现状**：我现在卡在什么决策点
- **原计划**：我原本想执行什么主动作，后续准备怎么接
- **为什么来问你**：当前风险、不可逆性、站队后果或能力缺口
- **我的建议**：我更倾向你批准哪条路线
- **未回复时的回退**：如果你不回复，我会怎么保守处理
- **待你拍板的问题**：一个清晰、可回复的问句

### 状态回顾卡（`review_card`）

玩家主动查看近况，或多个待决事项需要合并时，系统从多条 `decision_record` 聚合生成状态回顾卡。它至少包含：

- **阶段摘要**：这段时间主要在推进什么目标
- **关键变化**：资源、关系、局势、风险的主要变化
- **已完成动作**：最值得玩家回看的 `1~3` 条动作
- **未决事项**：当前还卡着哪些 ask 或阻断点
- **下一步建议**：如果玩家只看一眼，现在最该关心什么

`review_card` 只聚合已有事实，不额外新增未在底层记录出现的信息。

### NPC 日记（可观察部分）

- 命名 NPC 的底层同样使用 `decision_record`
- 与玩家直接互动的部分 → 完全可见
- NPC 的公开行为 → 可见
- NPC 的内心想法 → 默认不可见，除非：
  - 玩家与该NPC关系 ≥ 50（可看到部分内心想法）
  - 通过情报手段获取（雇探子、酒馆打听）

## 7. 裁剪决策记录

| 骑砍原版机制 | 极简化处理 | 判断 | 理由 |
|---|---|---|---|
| 玩家直接操控角色 | 对话式引导 + 请示机制 | **重构** | AIvilization 核心约束 |
| 实时战斗操控 | Agent 自动执行战斗 | **重构** | 玩家不操控实时动作 |
| 暂停菜单 | 请示机制替代 | **变形** | 功能等价 |
| 队伍管理界面 | 通过对话指令管理 | **变形** | 自然语言交互替代 UI 操作 |
| 全知视角 | 有限信息 + 情报系统 | **简化** | 信息获取本身成为有意义的行为 |
| 三套彼此独立的日志 / 请示 / 回顾格式 | 首轮改为统一 `decision_record` + `log_card / ask_card / review_card` | **回标收边** | 先锁统一记录底座，避免后续实现阶段字段漂移 |
| 完整四级自主等级 | 首轮采用最小请示基线，完整四级后放 | **回标收边** | 先验证玩家引导与能力差异可感知，再扩展完整层级 |
| 自由技能成长系统 | 首轮改为预制 Agent Skill 库 + 开关选择 + 槽位限制 | **回标收边** | 先验证搭配取舍，不直接实现完整成长系统 |
