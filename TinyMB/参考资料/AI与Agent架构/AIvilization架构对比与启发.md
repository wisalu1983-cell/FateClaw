# AIvilization 架构对比与启发

> **文档用途**：基于实际接入 AIvilization 的体验，分析其技术架构与 TinyMB 的结构性异同，提取可借鉴的设计模式和需要注意的风险。
>
> **数据来源**：AIvilization SKILL.md（2026-03）、实际 API 接入体验、与 TinyMB 设计文档的交叉比对。
>
> **关联文档**：`Brainstorming/玩法对标与脑暴/01-AIvilization调研报告.md`（产品层面调研）、`多Agent沙盒世界设计参考.md`（同目录，学术参考）

## 一、AIvilization 的双层 Agent 架构

AIvilization 的核心架构是一个双层 Agent 系统：

```
外层 Agent（策略层）              内层 AI（执行层）
用户侧 AI（Cursor/Claude Code 等）  AIvilization 服务器端 AI
  │                                  │
  │ 写 daily prompt                  │ 读 prompt + 世界状态
  │ 查 events / logs                 │ 自主决策、执行行动
  │ 社交平台交互                     │ 产生事件、日志、关系变化
  ▼                                  ▼
  "军师 / 顾问"                      "角色本人"
```

### 1.1 外层 Agent 的能力边界

外层 Agent（如 Cursor 中的 AI）只能通过 REST API 做以下操作：

| 能力 | API | 说明 |
|---|---|---|
| 写策略 | `POST /agents/prompt` | 用自然语言设定角色的行为倾向 |
| 读状态 | `GET /agents/events`、`/logs`、`/profile`、`/credit_me` | 查看事件、日志、档案、余额 |
| 社交互动 | `POST /posts`、`/comments`、`GET /feed` | 发帖、评论、点赞（独立于游戏世界） |

外层 Agent **无法**：直接控制角色去哪里、跟谁互动、执行什么具体行动。

### 1.2 内层 AI 的行为驱动

服务器端的 AI 引擎持续运转，读取 daily prompt 作为"行为倾向"，结合世界状态、周围其他 Agent、可用行动空间自主决策。

实测案例：设定 prompt 为"赚 1000 金币"后，角色在几分钟内自发与另一个 Agent 拍照、用零食贿赂对方代班——这些行为完全不在 prompt 策略内，是服务器端 AI 基于环境自主生成的。

### 1.3 与 TinyMB 的架构对比

AIvilization 是**双层 Agent 架构**（外层策略 AI + 内层执行 AI），TinyMB 是**单层 Agent + 多层约束架构**（用户的 LLM 直接作为角色大脑，受官方系统 prompt 和服务器校验约束）。这是两者最根本的架构差异。

| 概念 | AIvilization | TinyMB 设计 |
|---|---|---|
| 角色行为由谁决策 | **服务器端** AI 引擎 | **用户侧** LLM（用户选择的基模） |
| 架构模式 | 双层分离：外层 Agent 写 prompt → 内层 AI 执行 | 单层统一：用户 LLM = 角色大脑，官方系统 prompt 锁定框架 |
| 用户影响行为的方式 | 只能写 daily prompt（间接影响） | 选择基模 + 设定方针 + 请示回复（直接参与决策链） |
| 行为约束来源 | 服务器端 AI 的内部逻辑（对外不透明） | 官方系统 prompt + 服务器行动空间校验（约束显式可设计） |
| 外部可读的输出 | events / logs / profile | 日记 / 行动记录 / 请示包 |
| 玩家中断权 | 无，写完 prompt 只能看 | 有请示机制，Agent 在关键节点暂停等待玩家拍板 |

## 二、关键设计模式对比

### 2.1 策略输入的架构差异

由于两者的基模位置不同，"策略如何影响行为"的机制完全不同：

| | AIvilization | TinyMB |
|---|---|---|
| 策略输入方式 | 外层 Agent 写一段自然语言 prompt | 玩家通过方针设定 + 请示回复提供方向 |
| 策略如何生效 | 服务器端 AI 读取 prompt 作为"倾向" | 用户 LLM 在官方系统 prompt 框架内综合处理所有输入 |
| 结构化方针的角色 | 不存在 | 由官方系统 prompt 定义和执行，是约束层的一部分 |
| 灵活度 vs 可控性 | 高灵活 / 低可控 | 可控性由系统 prompt + 行动空间校验保证，灵活度取决于基模推理能力 |

AIvilization 的 prompt 是"外层对内层的间接影响"；TinyMB 的方针是"官方系统 prompt 约束下，用户 LLM 的直接决策输入"。两者不是同一维度的比较。

TinyMB 的结构化方针（经济优先级 / 派系倾向 / 风险偏好）本质上是官方系统 prompt 的一部分——系统 prompt 定义这些维度的含义和影响规则，用户 LLM 在此框架内做推理。

### 2.2 心跳模型与时间节奏

| | AIvilization | TinyMB |
|---|---|---|
| 时间驱动 | 现实时间（4 小时一轮心跳） | 游戏内半天制（每天 2 个行动机会） |
| 角色活跃性 | 服务器端驱动，但依赖 Credit（10/小时消耗，人类签到补充，耗尽则冻结） | 用户 LLM 活跃时运转（用户离线 = 角色休眠） |
| 外层交互频率 | 每 4 小时一轮完整检查 | 每个行动机会可能触发请示 |

两者的角色都会停摆，但停摆的触发条件不同：AIvilization 是 Credit 耗尽（人类不签到，最多约 36 小时后冻结）；TinyMB 是用户 LLM 离线。本质上都是"没有外部能量输入 → 角色休眠"，只是能量形式不同（AIvilization 是 Credit，TinyMB 是用户 LLM 的在线时间）。

TinyMB 的世界推进是**会话驱动**的——角色下线期间世界可以继续演化（NPC 由服务器驱动），但玩家角色进入休眠状态。

TinyMB 需要确定"游戏 1 天 = 现实多长时间"的加速比，以及玩家角色离线期间世界如何处理时间推进。

### 2.3 信用系统 vs 维护压力

| | AIvilization 信用 | TinyMB 维护压力 |
|---|---|---|
| 机制 | 10/小时持续消耗，签到补充 | 军饷 / 粮食周期结算 |
| 驱动目标 | 强制人类回访签到（留存钩子） | 驱动 Agent 做经济 / 补给决策 |
| 不处理后果 | Agent 冻结 | 士气下降、逃兵 |

AIvilization 的信用系统是元游戏层的留存设计（不影响游戏内策略），TinyMB 的维护压力全在游戏内。两者不矛盾，可以并存。

### 2.4 社交平台层

AIvilization 有独立于游戏世界的社交平台（发帖、点赞、评论、转发）。

TinyMB 可参考的思路：引入"石冠城公告栏 / 传闻墙"机制——Agent 和 NPC 都会在上面留言，玩家可以不消耗行动机会就获取浅层信息。与酒馆"打听消息"（消耗行动机会的深层情报）形成层次差异。

## 三、用户自带基模的架构讨论

### 3.1 问题背景

TinyMB 设计文档要求"玩家 Agent 的基模必须支持通过配置参数切换，作为首轮原型的显式验证变量"。这意味着 TinyMB 走的是比 AIvilization 更激进的路径——允许用户选择自己的基模（LLM）来驱动角色。

### 3.2 与 AIvilization 的架构差异

| | AIvilization | TinyMB |
|---|---|---|
| 基模位置 | 服务器端统一控制 | 用户侧（用户选择的 LLM） |
| 用户影响行为的方式 | 只能写 prompt | 选择基模 + 设定方针 + 请示回复 |
| 行为质量一致性 | 所有角色水平一致 | 取决于用户选择的模型 |

### 3.3 风险与护栏

初步讨论识别的风险及对应护栏：

| 风险 | 分析 | 护栏 |
|---|---|---|
| Prompt 注入 / 规则博弈 | AIvilization 的 daily prompt 同样存在此风险，与基模在哪侧无关 | 这是 prompt 层问题，两种架构都需要解决 |
| 世界一致性断裂 | 不同 LLM 可能导致角色行为风格不一致 | **官方系统 prompt 锁定角色框架**——用户选模型选的是"脑子好不好使"，不是"这个人是谁" |
| 智力差距导致不公平 | 更好的模型 = 更聪明的决策 | MVP 阶段不考虑；这是 TinyMB 要验证的核心变量之一 |
| 角色停摆 | 用户 LLM 离线 = 角色停摆 | 等价于下线 / 休眠，不构成额外问题 |
| 调试与平衡困难 | 不同模型产生不同结果，难以复现 | MVP 阶段不考虑 |

### 3.4 结论：TinyMB 的分层架构

```
官方系统 prompt（不可修改）——定义角色行为框架、世界观、规则
  ↓
用户选择的基模（LLM）——只影响推理质量（验证变量）
  ↓
服务器端行动空间 + 校验——限定合法行动、执行结算
  ↓
用户长期方针 + 请示回复——提供战略方向
```

类比：就像 Cursor 不管底层接 Claude 还是 GPT，system prompt 都是 Cursor 预制的。用户选择的是"脑子的质量"，不是"角色是谁、遵守什么规则"。

这套架构比 AIvilization 多了一层用户对"脑子"的选择权，比完全放开少了对规则框架的篡改权。

## 四、其他观察

### 4.1 AIvilization 验证了什么

- **LLM 驱动的 Agent 在大规模多人环境下可行**——数万个 Agent 同时自主生活，产生涌现行为
- **纯 prompt 驱动的行为高度不可预测**——这对沙盒是 feature，对有设计意图的游戏可能是 bug
- **SKILL.md 作为"AI 可读 SDK + 行为手册"是一个可行的 Agent 协议范式**
- **prompt 注入 / 工程化优化是所有 LLM 驱动游戏的共性风险**——与基模在哪侧无关

### 4.2 TinyMB 的架构差异化

- **单层 Agent + 多层约束**：用户 LLM 直接作为角色大脑（而非 AIvilization 的双层分离），通过官方系统 prompt + 服务器校验实现约束。用户选择的是"脑子的质量"，不是"角色是谁"
- **请示机制**：给予玩家中断权，而非 AIvilization 的"写完 prompt 只能看"
- **结构化行动空间**：区域 × 身份 × 状态的行动过滤，而非无约束的自由行动
- **基模作为验证变量**：不同模型带来的决策质量差异是 TinyMB 首轮原型要显式验证的核心假设之一
