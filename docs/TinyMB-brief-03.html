③ 增强二 · 参与

增强二 · 参与

如何在 Trumen_AgentCraft 的 MVP 中实现参与模式增强——两种模式的设计、请示机制、验证方案

设计思路

AIvilization 的问题
AIvilization 只有一种参与方式：设好策略 prompt，然后等待观察结果。玩家大部分时间只是旁观者，缺乏介入感。参与方式单一、频率固定、不可调节。

Trumen_AgentCraft 的解法：将参与方式从「只有低频放置」扩展为「高频到低频的完整频谱」——玩家可以实时深度介入，也可以挂机放置，两种模式随时切换。

预期体验

灵感来自于工作中使用 AI 编程助手（如 Claude Code、Cursor）的体验：

有时你和 AI 高频互动——提出想法、AI 执行反馈、遇到关键分歧时暂停等你拍板；有时你给 AI 一个宽泛目标让它自主完成，只在最后检查成果。

两种模式既需要底层能力支撑（系统架构上同时支持实时交互和自主运行），也给了玩家根据自己的需求主动选择的空间。

映射到游戏中：玩家在紧张的局势里和 Agent 密切对话、逐步推进；在日常阶段则设定方针后放手，让 Agent 自主行动——回来时翻看日志，发现世界已经发生了变化。

游戏中的情境举例

从实际使用 AI 编程助手的经验中，可以提炼出玩家选择高频互动的三种典型触发条件——它们在游戏中都有直接对应：

情境一：表达精度敏感——结果好坏需要逐句判断
编程类比：打磨对外文档措辞时，规则无法预定义"什么算好"，只能看到产出后逐句判断"对不对"。
游戏对应：外交谈判、拟定公告、起草贸易条款——Agent 提出措辞方案，玩家逐步审阅和调整，直到对方能接受。
情境二：思路渐进成型——方向在互动中逐步明确
编程类比：产品概念还模糊时，需要看到 agent 的产出后才能判断方向对不对，不断修正直到思路清晰。
游戏对应：探索未知局势、制定复合策略、应对多方博弈——三派系同时施压，Agent 提出初步方案，玩家在对话中不断调整策略方向。
情境三：体验驱动反馈——问题要做了才知道
编程类比：交互原型只有亲手试用后才能发现体验问题，反馈依赖即时感受。
游戏对应：任务执行中的实时决策、突发事件应对——Agent 执行护送任务途中遭遇伏击，玩家需要看到当前局面后即时判断：战斗、谈判还是撤退。
核心规律
当结果的好坏无法用明确规则预定义、必须靠人的主观判断来逐步逼近时，玩家会选择高频互动；当任务有明确目标和可验证标准时，玩家倾向于让 Agent 自主运行。
对照：方针放置模式
上述三种情境的反面——目标明确、路径清晰、判断标准可预设：「优先赚钱、保持中立、规避高风险」。Agent 按方针自主执行，玩家隔一段时间回来翻日志、看世界变化。
场景：睡前设定方针，隔天回来发现 Agent 已经在市场站稳了脚跟

MVP 中的具体落地

前面提到实时协作模式中，Agent 在关键节点会暂停请示、等玩家拍板——请示包就是这个机制的具体形态。当 Agent 遇到高后果决策点（明确站队、高风险行动、大额资源投入），它会暂停执行，向玩家提交一个包含当前局面、选项分析和风险预判的结构化决策包：

请示的设计意图
Agent 不是在所有决策上都打断玩家——日常事务（补给、移动、常规任务）自主执行后再汇报。只有在命中明确条件时才向玩家请示：明确站队或改变派系立场、高风险高后果行动、大额资源投入、发现自身能力不足以完成目标。请示包含当前局面、Agent 的建议、风险分析、替代路径，以及「如果你没回复我会怎么做」。

验证方式

验证假设
玩家会根据情境自主在实时协作和方针放置之间切换，而非被锁死在单一模式。参与频谱的两端都有自然的使用场景。
观察指标
玩家实际互动频率的分布；两种模式之间切换的频次和触发时机；请示包到达时的响应率和玩家决策质量（是否利用了 Agent 提供的信息）。
成功标准
两种模式均有自然使用场景，而非一种被废弃；请示包到达时玩家有足够信息做出有意义的决策；玩家不会因参与方式不灵活而流失。
