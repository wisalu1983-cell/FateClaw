PLUS③ · Agent能力差异

如何在 Trumen_AgentCraft 的 MVP 中实现 AI 差异增强——架构分工、基模选择机制、验证方案。

设计思路

三
Agent能力差异
✕ AIvilization 中所有 Agent 共用服务器端同一个模型，用户无感知、无选择权。

就像 OpenClaw 接入不同基模时表现出明显不同的对话质量和推理深度，Trumen_AgentCraft 中的玩家 Agent 同样由一个基础大语言模型（基模）直接驱动。基模决定的是 Agent 的智力水平——同样的任务指令，接入更强基模的 Agent 会做出更深入的局面分析、更灵活的路径规划、更合理的取舍判断。

这个差异在 Trumen_AgentCraft 中对玩家可感知：你能在请示包和行动日志里看到 Agent 怎么思考、为什么选这条路、在哪里判断力不足。

关键区分
基模不影响角色是谁、遵守什么规则（这些由官方系统 prompt 锁定），它影响的是「脑子好不好使」——推理质量、局面分析深度、决策合理性。

这是首轮原型要显式验证的核心变量：同等任务条件下，不同基模产生什么样的差异？

技术架构——用户侧与系统侧的分工

Trumen_AgentCraft 的多 Agent 框架采用三层结构，将角色智能与权威状态分离。Agent Framework 是与具体玩法解耦的角色决策与协作子系统——其底层运行逻辑与 OpenClaw 高度相似，运行环境是一个特定的游戏世界。

主信息流

输入 玩家输入链路
  1. Client 提交玩家输入（方针设定、请示回复、对话指令等）
  2. 输入进入玩家 Agent Framework 的决策上下文，随后走标准玩家 Agent 链路
玩家 玩家 Agent 链路
  1. Runtime → 用户侧玩家 Agent Framework：发送观察包
  2. 玩家 Agent Framework（用户基模 + 官方 prompt）→ Runtime：返回结构化行动意图
  3. Runtime：校验与结算流程与 NPC 链路完全相同，更新权威状态并同步 Client
NPC NPC Agent 链路
  1. Runtime → NPC Agent Framework：发送观察包
  2. NPC Agent Framework → Runtime：返回结构化行动意图
  3. Runtime：代码规则或 LLM 裁判完成校验与结算，更新权威状态并同步 Client

验证方式

验证假设
不同基模在同等任务条件下会产生玩家可感知的行为差异——推理质量的高低会体现在决策路径、取舍偏好和任务执行效率上。
观察指标
控制相同初始条件和角色人设，对比不同基模的 Agent 在决策路径、风险评估深度、多步规划合理性、取舍偏好上的表现差异。
成功标准
差异可被玩家观察到并形成偏好（"这个基模的 Agent 明显更会算账 / 更敢冒险"）；不同基模不会全部收敛到相同行为模式。

备注：MVP 多人场景下的 NPC 方案

LLM 驱动的 NPC 是 MVP 的加分项而非三项增强的验证前提——三项增强的验证目标均在玩家 Agent 侧。如果 MVP 需要支持多人，以下方案可避免"NPC 同时和多个玩家互动"的复杂度：

方案：NPC 公共状态统一 + 一对一按需互动
公共状态
NPC 位置、派系立场、市场价格、政策——全服统一一份，代码规则驱动变化，不需要 LLM
私有状态
NPC 和每个玩家的关系值、对话历史、任务进度——按玩家隔离，互不干扰
LLM 调用
仅在玩家发起一对一互动时按需触发。NPC 结合公共状态 + 和该玩家的私有状态来响应
并发处理
多个玩家同时找同一个 NPC → 排队处理，和 MMO NPC 交互队列一致
世界演化
取消 NPC 间自主互动，世界变化由规则系统驱动（价格波动、派系张力、定时事件）

本质上是 MMO NPC 经典模型 + LLM 对话能力：NPC 的存在和公共行为走传统方案，交互时用 LLM 生成自然语言对话和推理判断。服务端 LLM 成本从"NPC 数 × tick 频率"降为"玩家互动次数"，按需付费。
