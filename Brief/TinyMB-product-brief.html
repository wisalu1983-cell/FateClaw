Trumen · Phase 1 · 产品构思

TinyMB

Trumen 第一阶段验证载体。以极简化《骑马与砍杀》城镇玩法为规则骨架， 在持续运转的单城邦中让多个 LLM 驱动的 Agent 自主行动与交互。

AIvilization Plus
一

第一阶段要做什么

定位

TinyMB 是 AIvilization 的 Plus 版本，也是 Trumen 第一阶段（P1 世界模拟）的验证载体。
AIvilization 验证了基础命题——LLM 驱动的 Agent 能在共享世界中自主生活——但留下了三个空白。TinyMB 逐一填补：
▲ AIvilization 的空白 ✦ TinyMB 的增强
规则 无结构沙盒，行动无代价可衡量 极简骑砍规则：三派系博弈 + 四变量取舍 + 维护压力
参与 只能低频写 prompt 然后旁观，参与度低 高频实时对话 ↔ 低频挂机放置，完整频谱随时切换
AI 差异 统一服务器模型，用户无感 玩家自选基模，差异可感知、可对比、可验证

产品形态

AIvilization 的底座保留：一个持续运转的共享世界，Agent 有自己的生活节奏，多个 Agent 之间产生涌现行为。TinyMB 在这个底座上叠加三项增强。

一
有规则的活世界
✕ AIvilization 的行动没有结构化后果，Agent 的决策好坏无从衡量。

一座名为「石冠城」的自治城邦持续运转。三大派系——守备军团、商盟、平民公社——围绕税收、粮价、安全互相牵制。6 名有独立性格和目标的命名 NPC 各自行动：在城主堡巡视、在市集做买卖、在酒馆交换情报、在农庄护理民生。

每个行动都挂着代价：帮一方就会得罪另一方，接了高回报的灰色交易就要承担荣誉下降的后果，随从要吃饭要发饷，不维护就会流失。金钱、声望、荣誉、关系四个变量构成持续的取舍压力。

二
引导而非控制
✕ AIvilization 只有一种节奏——每隔几小时写 prompt 然后等，玩家大部分时间只是旁观者。

TinyMB 把参与频谱从「只有低频」扩展为「低频到高频都有」，两种模式随时切换：

高频模式（类似用 Claude Code）
玩家和 Agent 实时对话。每步行动 10-30 秒一轮反馈，遇到重大决策 Agent 暂停请示，玩家审查后拍板。
低频模式（类似 AIvilization 放置）
设定长期方针（经济优先级 / 派系倾向 / 风险偏好），Agent 自主行动，玩家隔一段时间回来翻日志。
三
AI 脑子可选可感知
✕ AIvilization 中所有 Agent 共用服务器端同一个模型，用户无感知、无选择权。

玩家可以选择驱动自己 Agent 的基础大语言模型（基模）。基模决定的是「脑子好不好使」——推理深度、规划质量、应变能力——而不是角色是谁、遵守什么规则（这些由官方系统 prompt 锁定）。

同一任务下，不同基模会产出不同的执行规划和路径，这个差异对玩家可感知，也是首轮原型要显式验证的核心变量。

预期的游戏体验

继承自 AIvilization 的底色
世界不因玩家下线而暂停，NPC 有自己的节奏，多个 Agent 之间产生超出预设的涌现行为。玩家上线时发现世界已经变了——有人暗中交换了情报，派系力量对比悄然移动。这种「世界活着」的感受是 TinyMB 继承的核心底色。
三项增强带来的新体验
增强体验变化
有规则的活世界 不是「看 Agent 做事」，而是「承担后果、据此调整方向」——帮了守备军团可能得罪商盟，接了灰色交易可能失去正直 NPC 的信任
引导而非控制 参与深度由玩家自己控制：想深度介入就实时对话（10-30 秒一轮反馈 + 关键节点拍板），没空就挂机方针驱动
AI 脑子可选 换个基模，同样的任务可能有明显不同的执行路径和结果——差异可观察，是首轮验证要回答的核心问题
◆ 一句话总结： AIvilization 的活世界底色 + 骑砍式规则带来的有后果取舍 + 从实时对话到挂机放置的完整参与频谱 + 基模选择带来的可感知 AI 差异。
二

三项增强的设计展开

以下展开说明上述三项增强在设计层面的具体变化。每一项都直接服务 P1「证明世界能跑起来、Agent 能自主行动并交互」的使命。

增强一 有规则的活世界——从无结构沙盒到有规则博弈
维度AIvilizationTinyMB
玩法骨架 无预设目标和博弈结构，Agent 自由行动 骑砍四变量（金钱 / 声望 / 荣誉 / 关系）+ 三派系博弈 + 维护压力
行动后果 后果分散，无结构化取舍 帮一方得罪另一方、资源有限必须取舍、站队有不可逆后果
空间约束 无移动成本 区域间移动耗时 5-20 分钟，迫使规划连续事务
为什么改
无规则意味着 Agent 的决策好坏无从衡量。引入结构化规则后，Agent 的行为有了可评估的标准——「在有限资源和派系压力下做出什么取舍」。
服务 P1 的方式
结构化规则让 Agent 行为可评估、可比较、可复现，为「Agent 能自主行动并交互」提供可验证的观察框架。
增强二 引导而非控制——从「只有低频」到「低频高频都有」

AIvilization 的用户参与度低，根本原因是架构只支持一种节奏：写 prompt → 等结果。TinyMB 同时改变了底层架构和上层交互方式。

AIvilization · 双层 Agent
外层 AI（Cursor / Claude Code）
写每日策略 prompt
服务器端 AI 引擎
读 prompt → 自主决策执行
用户只能写 prompt 和看日志
无法中断或实时参与
→
TinyMB · 单层 Agent + 多层约束
用户选择的 LLM（基模）
直接作为角色大脑，用户侧运行
官方系统 prompt（不可修改）
锁定行为框架与规则
服务器 Runtime
校验合法行动、结算结果
维度AIvilization（双层 Agent）TinyMB（单层 Agent）
行为决策者 服务器端 AI 引擎 用户选择的 LLM（基模）
用户影响方式 只能写每日策略 prompt（间接影响） 对话引导 + 请示回复 + 长期方针（直接参与决策链）
玩家中断权 无——写完 prompt 只能看结果 有——Agent 在关键节点暂停请示，玩家拍板
行为约束来源 服务器端内部逻辑（对外不透明） 官方系统 prompt（显式可设计）+ 服务器行动空间校验
AIvilization 像是你写了一封策略建议书交给一个你看不到的经理人，他按自己的理解去执行；TinyMB 像是你直接坐在执行者旁边，他按规矩办事但遇到大事会转过头来问你——当然你也可以起身离开，他会按你之前定好的方针继续干活。
为什么改
双层架构下参与方式被锁死在低频放置。单层架构打开了高频实时互动的可能——每轮行动 10-30 秒即有反馈，同时保留了长期方针驱动的低频放置模式，形成完整的参与频谱。
服务 P1 的方式
高频互动提供可观察的行动日志和请示闭环；低频放置验证 Agent 在无人干预下的自主决策质量。
增强三 AI 脑子可选可感知——基模作为显式验证变量
维度AIvilizationTinyMB
基模位置 服务器端统一控制，用户无感 用户侧，用户选择基模
模型差异的作用 不存在——所有角色用同一个模型 是首轮要验证的核心假设之一
为什么改
Trumen 的长期愿景需要验证「不同 AI 能力会带来不同的世界演化」。TinyMB 将基模选择权交给用户，使「同一任务条件下，不同基模产生不同执行路径和结果」成为首轮可观测、可对比的验证变量。
服务 P1 的方式
为「多 Agent 框架」提供可对比的能力维度——不同基模在同样的规则和约束下表现如何，是 P1 技术可行性验证的关键数据点。

世界涌现

P2 的使命是「让创作者能按意图发起属于自己的世界」。TinyMB 在 P1 跑通的那套东西，意图是将来能被拆出来、配置化，成为 P2「世界发起工具」的底层能力。

TinyMB（P1）产出物 对 P2 的意图
统一的 Agent 行为循环 P2 需要放入 Agent 并让它们自主行动，P1 的循环框架是这一能力的基础
Agent 与 Runtime 的行动协议 P2 发起的世界需要能校验和执行 Agent 行动，P1 定义了这个接口应该长什么样
世界信息向 Agent 的传递结构 P2 的不同世界需要向 Agent 传递环境信息，P1 验证了这套结构的必要形态
可观察性记录机制 P2 的世界需要能被创作者回看和运营，P1 的记录机制是这一能力的前提
结构化的世界规则集 P2 需要让创作者配置和组合规则，P1 的规则集是第一个完整跑通的样本
基模差异的实验数据 P2 需要支持 Agent 能力的配置，P1 的数据帮助确定能力维度的边界
附

关键术语速查

基模
驱动 Agent 核心推理的基础大语言模型（LLM），用户可选择，影响推理质量但不影响角色身份与规则框架
Agent Skill
Agent 处理特定类型事务的任务能力包（方法、判断框架、流程经验），不同于传统 RPG 技能树
四变量
金钱 / 声望 / 荣誉 / 关系——源自骑砍的核心策略维度
请示
Agent 在关键决策点暂停执行、向玩家提交决策包等待审批的机制
Runtime
服务器端的行动校验与结算引擎，是所有 Agent 行动的权威执行者
命名 NPC
有唯一名字、性格、目标和派系身份的完整 Agent 实体，由服务端 LLM 驱动
